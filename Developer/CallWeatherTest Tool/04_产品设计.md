# Weather API 调用测试工具 - 产品设计

## 1. 产品定位

### 1.1 产品愿景

成为 Developer 在 API 平台进行接口调试的首选工具，以极简的操作流程和清晰的展示方式提升开发效率。

### 1.2 目标用户画像

**用户 A：小明（初级开发者）**
- 背景：计算机专业学生，刚接触 API 开发
- 痛点：不知道如何验证获得的 API Key 是否有效
- 期望：简单三步完成测试

**用户 B：李工（测试工程师）**
- 背景：负责 API 平台的质量保障
- 痛点：需要批量测试不同参数的返回结果
- 期望：能快速切换参数，查看格式化响应

**用户 C：王姐（集成开发者）**
- 背景：正在将天气 API 集成到自己的应用
- 痛点：需要确认响应格式以便编写解析代码
- 期望：直接复制 JSON 格式的响应示例

## 4. 交互设计

### 4.1 用户交互

| 交互 | 触发 | 反馈 |
|------|------|------|
| 保存 API Key | 点击「保存」按钮 | Toast: "配置已保存" |
| 发送请求 | 点击「发送请求」 | 按钮显示 loading，禁用输入 |
| 复制响应 | 点击「复制」按钮 | Toast: "已复制到剪贴板" |
| 清空历史 | 点击「清空全部」 | 确认对话框，确认后清空 |
| 选中历史项 | 点击历史记录 | 填充对应请求参数 |

### 4.2 快捷键

| 快捷键 | 功能 |
|--------|------|
| Ctrl + Enter | 发送请求 |
| Ctrl + C (响应区) | 复制响应内容 |
| Esc | 关闭弹窗 / 清空输入 |

## 5. 数据设计

### 5.1 LocalStorage 数据结构

```javascript
// Key: weather_api_config
{
  apiKey: "sk-xxx...xxx",
  apiUrl: "http://localhost:8000/api/v1",
  savedAt: "2026-04-21T18:00:00Z"
}

// Key: weather_call_history
[
  {
    id: "uuid-v4",
    timestamp: "2026-04-21T18:00:00Z",
    method: "GET",
    endpoint: "/weather/current",
    params: { city: "北京" },
    status: 200,
    duration: 156,  // ms
    response: { ... }
  }
]
```

### 5.2 数据校验

| 字段 | 校验规则 |
|------|----------|
| API Key | 非空，长度 > 10 |
| API URL | 有效的 HTTP/HTTPS URL |
| 城市名称 | 非空，2-20 个字符 |
| 历史记录 | 最多 10 条，自动淘汰最旧记录 |

## 6. 错误处理

### 6.1 错误展示

```
┌─────────────────────────────────────────────────────┐
│  ❌ 请求失败                                          │
│  ─────────────────────────────────────────────────  │
│  错误码: 401                                         │
│  错误信息: Unauthorized                              │
│  详情: API Key 无效或已过期                          │
│                                                     │
│  [重新尝试]                                          │
└─────────────────────────────────────────────────────┘
```

### 6.2 错误码对照

| 错误码 | 错误信息 | 解决方案 |
|--------|----------|----------|
| 401 | Unauthorized | 检查 API Key 是否正确 |
| 403 | Forbidden | Key 无权访问此接口 |
| 404 | Not Found | 检查 API 地址是否正确 |
| 429 | Too Many Requests | 降低请求频率 |
| 500 | Internal Server Error | 联系技术支持 |
| - | Network Error | 检查网络连接 |

## 7. 性能设计

### 7.1 性能指标

| 指标 | 目标 | 测量方法 |
|------|------|----------|
| 首次加载 | < 2s | Lighthouse |
| 请求响应 | < 30s 超时 | 用户感知 |
| 界面响应 | < 100ms | 用户感知 |

### 7.2 优化策略

- CSS/JS 使用 CDN 加速
- 图片资源懒加载
- 响应内容虚拟滚动（大量数据时）
- 请求防抖（300ms）
