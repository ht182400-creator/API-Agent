# API 测试工具设计方案

## 1. 需求规格

### 1.1 背景

随着 API 平台仓库数量的增长，Developer 测试工具需要支持更多 API 仓库。当前硬编码的方式不便于扩展，需要设计一个可配置的、易于扩展的解决方案。

### 1.2 目标

- **统一入口**：一个测试工具支持所有仓库的 API 测试
- **配置驱动**：通过配置文件管理仓库和端点，无需修改代码
- **易于扩展**：新增仓库只需添加配置文件
- **分类清晰**：支持自研仓库和第三方仓库的分类展示

### 1.3 功能需求

| 需求编号 | 需求描述 | 优先级 |
|----------|----------|--------|
| F-001 | 支持多仓库切换和展示 | P0 |
| F-002 | 支持仓库分类筛选（全部/自研/第三方） | P0 |
| F-003 | 动态表单生成（根据参数配置自动渲染） | P0 |
| F-004 | 支持多种 HTTP 方法（GET/POST/PUT/DELETE/PATCH） | P0 |
| F-005 | 响应结果展示和 curl 命令生成 | P0 |
| F-006 | 请求历史记录 | P1 |
| F-007 | 响应示例展示 | P1 |

### 1.4 非功能需求

| 需求编号 | 需求描述 | 优先级 |
|----------|----------|--------|
| N-001 | 页面加载时间 < 2s | P1 |
| N-002 | 请求响应时间展示 | P1 |
| N-003 | 错误信息友好提示 | P1 |
| N-004 | 支持键盘快捷键（Ctrl+Enter 发送） | P2 |

## 3. 数据模型

### 3.1 类型定义

```typescript
// 仓库定义
interface Repository {
  id: string;                    // 唯一标识
  slug: string;                  // URL标识
  name: string;                  // 显示名称
  description: string;           // 描述
  icon: string;                  // 图标
  category: 'self' | 'third_party';  // 分类
  baseUrl: string;               // 基础URL
  endpoints: Endpoint[];        // 端点列表
  authType: AuthType;            // 认证方式
  authHeader?: string;           // 认证头
  enabled?: boolean;             // 是否启用
  version?: string;             // 版本号
}

// 端点定义
interface Endpoint {
  id: string;                    // 端点ID
  name: string;                  // 名称
  description?: string;          // 描述
  path: string;                  // 路径
  method: HttpMethod;            // HTTP方法
  params?: Parameter[];          // 查询参数
  requestBody?: Parameter[];      // 请求体参数
  responseExample?: object;      // 响应示例
  tags?: string[];               // 标签（用于分组）
}

// 参数定义
interface Parameter {
  name: string;                  // 参数名
  type: ParamType;               // 类型
  required: boolean;             // 是否必填
  description: string;           // 描述
  placeholder?: string;          // 占位符
  defaultValue?: any;            // 默认值
  options?: {label, value}[];    // 选项（select类型）
  in: ParamLocation;             // 位置
  validation?: Validation;       // 验证规则
}
```

## 4. 对比分析

### 4.1 设计方案对比

| 特性 | 当前设计 | 新设计方案 |
|------|----------|------------|
| **添加新仓库** | 修改 app.js 代码 | **仅添加配置文件** |
| **API端点管理** | 硬编码函数 | **配置声明** |
| **表单生成** | 固定HTML表单 | **动态渲染** |
| **仓库分类** | 无 | **支持自研/第三方** |
| **请求历史** | localStorage | **支持，可回放** |
| **扩展难度** | 高（需改代码） | **低（添加配置）** |
| **维护成本** | 高 | **低** |

### 4.2 技术选型对比

| 维度 | 原生HTML方案 | React组件方案 |
|------|-------------|---------------|
| **集成方式** | 独立HTML文件 | **集成到平台前端** |
| **状态管理** | localStorage | **React State** |
| **路由管理** | 无 | **React Router** |
| **类型安全** | 无 | **TypeScript** |
| **代码复用** | 困难 | **组件化复用** |
| **开发效率** | 低 | **高** |
| **打包体积** | 无影响 | **需引入React** |

### 4.3 仓库配置对比

| 仓库 | 原方案支持 | 新方案支持 | 说明 |
|------|-----------|-----------|------|
| Weather API | ✅ | ✅ | 自研天气服务 |
| Logistics API | ❌ | ✅ | 可快速添加 |
| SMS API | ❌ | ✅ | 第三方短信 |
| Payment API | ❌ | ✅ | 第三方支付 |
| 更多仓库... | ❌ | ✅ | 按需添加 |
