## 1. 回滚策略

### 1.1 回滚原则

```
┌─────────────────────────────────────────────────────────────┐
│                       回滚原则                                  │
├─────────────────────────────────────────────────────────────┤
│                                                                          │
│  1. 快速恢复服务                                                │
│     - 优先恢复服务可用性                                         │
│     - 再排查问题根因                                            │
│                                                                          │
│  2. 数据一致性                                                  │
│     - 确保回滚后数据不会损坏                                     │
│     - 考虑数据库迁移的回滚                                        │
│                                                                          │
│  3. 沟通透明                                                    │
│     - 及时通知相关人员                                            │
│     - 向用户公告维护信息                                          │
│                                                                          │
│  4. 事后复盘                                                    │
│     - 记录问题原因                                                │
│     - 更新回滚预案                                                │
│                                                                          │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 回滚级别

| 级别 | 影响范围 | 回滚时间目标 | 预案 |
|------|---------|------------|------|
| **L1 - 前端静态资源** | 仅前端展示 | < 5分钟 | 保留历史版本，快速切换 |
| **L2 - 后端API** | API服务 | < 10分钟 | 蓝绿部署，快速切换 |
| **L3 - 数据库迁移** | 数据层 | < 30分钟 | 迁移脚本支持回滚 |
| **L4 - 重大故障** | 整体服务 | < 1小时 | 切换到备用环境 |

## 2. 回滚触发条件

### 2.1 自动触发条件

| 监控指标 | 阈值 | 持续时间 | 动作 |
|---------|--------|------------|------|
| **错误率** | > 5% | 5分钟 | 自动回滚（可选） |
| **API响应时间** | p95 > 2秒 | 10分钟 | 告警 + 人工决策 |
| **服务可用性** | < 95% | 3分钟 | 自动回滚 |
| **前端性能** | LCP > 5秒 | 5分钟 | 告警 + 人工决策 |

### 2.2 手动触发条件

```
触发场景：
1. 用户大量投诉功能异常
2. 关键业务流程中断（登录、支付等）
3. 数据丢失或错误
4. 安全漏洞暴露
5. 依赖服务（如第三方API）故障
```

### 2.3 不触发回滚的场景

```
不建议回滚的场景：
1. 样式轻微错位（不影响功能）
2. 非关键页面的小BUG
3. 个别浏览器兼容性问题
4. 文档错误
```

## 3. 回滚方案

### 3.1 方案一：保留历史版本（推荐）

**原理**：部署时保留旧版本，回滚时直接切换符号链接

**部署结构**：

```
/var/www/api-platform/
├── current -> releases/v1.2.3  # 符号链接，指向当前版本
├── releases/
│   ├── v1.2.3/             # 当前版本
│   ├── v1.2.2/             # 上一版本
│   ├── v1.2.1/             # 上上个版本
│   └── ...
└── shared/                  # 共享文件（日志、上传等）
```

**部署脚本**：

**文件**：`scripts/deploy.sh`

```bash
#!/bin/bash
set -e

# 配置
DEPLOY_DIR="/var/www/api-platform"
RELEASE_DIR="$DEPLOY_DIR/releases"
CURRENT_LINK="$DEPLOY_DIR/current"
KEEP_RELEASES=5  # 保留最近5个版本

# 获取版本号
VERSION=$(date +%Y%m%d_%H%M%S)
RELEASE_PATH="$RELEASE_DIR/$VERSION"

echo "开始部署版本：$VERSION"

# 步骤1：创建发布目录
mkdir -p "$RELEASE_PATH"

# 步骤2：复制构建产物
echo "复制构建产物..."
cp -r dist/* "$RELEASE_PATH/"

# 步骤3：复制共享文件
ln -s "$DEPLOY_DIR/shared/logs" "$RELEASE_PATH/logs"
ln -s "$DEPLOY_DIR/shared/uploads" "$RELEASE_PATH/uploads"

# 步骤4：切换符号链接（原子操作）
echo "切换到新版本..."
ln -sfn "$RELEASE_PATH" "$DEPLOY_DIR/current_new"
mv -Tf "$DEPLOY_DIR/current_new" "$CURRENT_LINK"

# 步骤5：重新加载Nginx
echo "重新加载Nginx..."
sudo nginx -t && sudo systemctl reload nginx

# 步骤6：清理旧版本
echo "清理旧版本..."
cd "$RELEASE_DIR"
ls -dt * | tail -n +$((KEEP_RELEASES + 1)) | xargs -r rm -rf

echo "部署完成！版本：$VERSION"
```

### 3.2 方案二：Git回滚

**原理**：通过Git回退代码，重新构建部署

**适用场景**：紧急修复、快速回滚到任意历史版本

**回滚脚本**：

**文件**：`scripts/rollback-git.sh`

```bash
#!/bin/bash
set -e

# 配置
TARGET_VERSION=$1  # 目标版本（Git commit ID或tag）

if [ -z "$TARGET_VERSION" ]; then
  echo "用法：$0 <git-commit-id>"
  exit 1
fi

echo "开始回滚到版本：$TARGET_VERSION"

# 步骤1：切换到目标版本
echo "切换Git分支..."
git checkout "$TARGET_VERSION"

# 步骤2：安装依赖
echo "安装依赖..."
npm ci

# 步骤3：构建
echo "构建项目..."
npm run build:production

# 步骤4：部署
echo "部署..."
sudo systemctl stop nginx
cp -r dist/* /var/www/api-platform/
sudo systemctl start nginx

echo "回滚完成！"
```

### 3.3 方案三：Docker镜像回滚

**原理**：使用Docker部署时，直接切换到旧版本镜像

**回滚脚本**：

**文件**：`scripts/rollback-docker.sh`

```bash
#!/bin/bash
set -e

# 配置
TARGET_VERSION=$1  # 目标版本（Docker镜像tag）

if [ -z "$TARGET_VERSION" ]; then
  echo "用法：$0 <version-tag>"
  echo "例如：$0 v1.2.2"
  exit 1
fi

echo "开始回滚到版本：$TARGET_VERSION"

# 步骤1：停止当前容器
echo "停止当前容器..."
docker stop api-platform-web || true

# 步骤2：删除当前容器
echo "删除当前容器..."
docker rm api-platform-web || true

# 步骤3：启动旧版本容器
echo "启动旧版本容器..."
docker run -d \
  --name api-platform-web \
  -p 3000:80 \
  --restart unless-stopped \
  api-platform-web:$TARGET_VERSION

echo "回滚完成！"
echo "查看日志：docker logs -f api-platform-web"
```

### 3.4 方案四：Blue-Green部署（零停机）

**架构**：

```
┌─────────────────────────────────────────────────────────────┐
│                   Blue-Green 部署                             │
├─────────────────────────────────────────────────────────────┤
│                                                                          │
│  负载均衡 (Nginx)                                               │
│       │                                                            │
│       ├──→ Blue环境 (v1.2.3) ← 当前生产环境                  │
│       │                                                            │
│       └──→ Green环境 (v1.2.4) ← 新版本（预热中）             │
│                                                                          │
│  回滚：切换负载均衡指向 Blue                                     │
│                                                                          │
└─────────────────────────────────────────────────────────────┘
```

**实现脚本**：

**文件**：`scripts/blue-green-deploy.sh`

```bash
#!/bin/bash
set -e

# 配置
NGINX_CONF="/etc/nginx/sites-available/api-platform"
BLUE_PORT=3001
GREEN_PORT=3002
CURRENT_ENV_FILE="/tmp/current-env"  # 记录当前环境

# 读取当前环境
if [ -f "$CURRENT_ENV_FILE" ]; then
  CURRENT_ENV=$(cat "$CURRENT_ENV_FILE")
else
  CURRENT_ENV="blue"
  echo "$CURRENT_ENV" > "$CURRENT_ENV_FILE"
fi

# 确定新环境
if [ "$CURRENT_ENV" == "blue" ]; then
  NEW_ENV="green"
  NEW_PORT=$GREEN_PORT
else
  NEW_ENV="blue"
  NEW_PORT=$BLUE_PORT
fi

echo "当前环境：$CURRENT_ENV"
echo "部署新版本到：$NEW_ENV (端口 $NEW_PORT)"

# 步骤1：停止新环境容器（如果存在）
docker stop api-platform-$NEW_ENV || true
docker rm api-platform-$NEW_ENV || true

# 步骤2：启动新环境容器
echo "启动 $NEW_ENV 环境..."
docker run -d \
  --name api-platform-$NEW_ENV \
  -p $NEW_PORT:80 \
  api-platform-web:latest

# 步骤3：等待新环境就绪
echo "等待 $NEW_ENV 环境就绪..."
for i in {1..30}; do
  if curl -s http://localhost:$NEW_PORT/health > /dev/null; then
    echo "新环境就绪！"
    break
  fi
  if [ $i -eq 30 ]; then
    echo "新环境启动超时！"
    exit 1
  fi
  sleep 2
done

# 步骤4：切换Nginx配置
echo "切换到 $NEW_ENV 环境..."
if [ "$NEW_ENV" == "green" ]; then
  sed -i 's/proxy_pass http:\/\/localhost:'$BLUE_PORT'/proxy_pass http:\/\/localhost:'$GREEN_PORT'/' "$NGINX_CONF"
else
  sed -i 's/proxy_pass http:\/\/localhost:'$GREEN_PORT'/proxy_pass http:\/\/localhost:'$BLUE_PORT'/' "$NGINX_CONF"
fi

# 步骤5：测试Nginx配置
echo "测试Nginx配置..."
sudo nginx -t

# 步骤6：重新加载Nginx
echo "重新加载Nginx..."
sudo systemctl reload nginx

# 步骤7：更新当前环境记录
echo "$NEW_ENV" > "$CURRENT_ENV_FILE"

echo "部署完成！当前环境：$NEW_ENV"

# 步骤8：回滚步骤（如果新版本有问题）
echo ""
echo "如果新版本有问题，执行以下命令回滚："
echo "  sed -i 's/proxy_pass http:\/\/localhost:'$NEW_PORT'/proxy_pass http:\/\/localhost:'$((NEW_ENV == "green") && echo $BLUE_PORT || echo $GREEN_PORT)'/' "$NGINX_CONF""
echo "  sudo systemctl reload nginx"
```

## 5. 回滚验证

### 5.1 自动化验证脚本

**文件**：`scripts/verify-rollback.sh`

```bash
#!/bin/bash

# 配置
BASE_URL=${1:-"http://localhost:3000"}
FAILED=0

echo "开始验证回滚结果：$BASE_URL"

# 测试1：健康检查
echo "测试1：健康检查..."
HEALTH_STATUS=$(curl -s -o /dev/null -w "%{http_code}" "$BASE_URL/health")
if [ "$HEALTH_STATUS" == "200" ]; then
  echo "✅ 健康检查通过"
else
  echo "❌ 健康检查失败：HTTP $HEALTH_STATUS"
  FAILED=1
fi

# 测试2：首页可访问
echo "测试2：首页可访问..."
HOME_STATUS=$(curl -s -o /dev/null -w "%{http_code}" "$BASE_URL/")
if [ "$HOME_STATUS" == "200" ]; then
  echo "✅ 首页可访问"
else
  echo "❌ 首页不可访问：HTTP $HOME_STATUS"
  FAILED=1
fi

# 测试3：静态资源可访问
echo "测试3：静态资源可访问..."
JS_STATUS=$(curl -s -o /dev/null -w "%{http_code}" "$BASE_URL/assets/js/index.js")
if [ "$JS_STATUS" == "200" ]; then
  echo "✅ 静态资源可访问"
else
  echo "❌ 静态资源不可访问：HTTP $JS_STATUS"
  FAILED=1
fi

# 测试4：API可访问（如果需要）
echo "测试4：API可访问..."
API_STATUS=$(curl -s -o /dev/null -w "%{http_code}" "$BASE_URL/api/health")
if [ "$API_STATUS" == "200" ]; then
  echo "✅ API可访问"
else
  echo "⚠️ API不可访问：HTTP $API_STATUS（如果API是独立的，可以忽略）"
fi

# 测试结果
echo ""
if [ $FAILED -eq 0 ]; then
  echo "🎉 所有验证通过！回滚成功！"
  exit 0
else
  echo "⚠️ 验证失败！请检查服务状态。"
  exit 1
fi
```

## 7. 应急预案

### 7.1 极端情况应对

**场景1：回滚失败**

```
应对步骤：
1. 保持冷静，不要慌张
2. 检查回滚脚本是否有误
3. 尝试手动回滚（详见7.2）
4. 如果仍然失败，启动备用环境
5. 通知所有相关人员
```

**场景2：数据库迁移无法回滚**

```
应对步骤：
1. 停止应用（避免数据进一步损坏）
2. 从备份恢复数据库
3. 切换到旧版本应用（兼容旧数据库结构）
4. 验证数据完整性
5. 公告用户（数据恢复中）
```

**场景3：CDN缓存导致旧版本无法立即生效**

```
应对步骤：
1. 登录CDN控制台
2. 刷新缓存（全站刷新 or 目录刷新）
3. 如果刷新太慢，修改CDN域名指向源站
4. 等待DNS生效（TTL）
5. 验证CDN是否返回正确版本
```
