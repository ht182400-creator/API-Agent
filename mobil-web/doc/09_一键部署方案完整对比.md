### 三种一键部署方案

| 方案 | 核心思想 | 适用场景 |
|------|---------|---------|
| **方案一: Docker Compose** | 容器编排，一键启动所有服务 | 本地开发、测试环境、小型生产环境 |
| **方案二: CI/CD自动化部署** | Git推送触发自动构建部署 | 中大型生产环境、团队协作 |
| **方案三: Electron安装包** | 桌面应用包装，双击安装 | 客户私有化部署、离线环境 |

## 📝 方案详解

### 方案一: Docker Compose 一键部署

#### 架构图

```
┌─────────────────────────────────────────────────────────────┐
│                     Docker Compose 架构                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [docker-compose.yml]                                      │
│          │                                                  │
│          ▼                                                  │
│   ┌─────────────────────────────────────────────────┐      │
│   │              Docker Network                      │      │
│   ├─────────────────────────────────────────────────┤      │
│   │                                                 │      │
│   │  ┌─────────────┐  ┌─────────────┐             │      │
│   │  │   Frontend   │  │   Backend   │             │      │
│   │  │   (Nginx)    │  │  (FastAPI)  │             │      │
│   │  │  Port: 80    │  │  Port: 8000 │             │      │
│   │  └─────────────┘  └─────────────┘             │      │
│   │          │                │                     │      │
│   │          ▼                ▼                     │      │
│   │  ┌─────────────┐  ┌─────────────┐             │      │
│   │  │ PostgreSQL   │  │    Redis    │             │      │
│   │  │ Port: 5432   │  │ Port: 6379  │             │      │
│   │  └─────────────┘  └─────────────┘             │      │
│   │          │                                    │      │
│   │          ▼                                    │      │
│   │  ┌─────────────┐                            │      │
│   │  │    MinIO     │                            │      │
│   │  │ Port: 9000   │                            │      │
│   │  └─────────────┘                            │      │
│   │                                                 │      │
│   └─────────────────────────────────────────────────┘      │
│                                                             │
│   一键启动命令:                                             │
│   docker-compose up -d                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 优点

1. **部署极简**: 一条命令启动所有服务
2. **环境隔离**: 容器化部署，不污染宿主机
3. **快速回滚**: 切换镜像版本即可
4. **资源可控**: 可限制每个容器的CPU、内存
5. **跨平台**: 支持 Windows、Mac、Linux

#### 缺点

1. **单点故障**: 所有服务在同一台机器
2. **扩展性有限**: 难以实现水平扩展
3. **网络复杂**: 容器网络配置较复杂
4. **存储持久化**: 需要小心配置数据卷

#### 实施步骤

```bash
# 1. 安装 Docker Desktop (Windows/Mac) 或 Docker Engine (Linux)
# Windows: https://docs.docker.com/desktop/install/windows-install/
# Mac: https://docs.docker.com/desktop/install/mac-install/
# Linux: https://docs.docker.com/engine/install/

# 2. 克隆代码
git clone https://github.com/your-repo/api-platform.git
cd api-platform

# 3. 配置环境变量
cp .env.example .env
# 编辑 .env 文件，修改密码等配置

# 4. 一键启动
docker-compose up -d

# 5. 查看服务状态
docker-compose ps

# 6. 查看日志
docker-compose logs -f

# 7. 停止服务
docker-compose down

# 8. 停止并删除数据卷 (慎用!)
docker-compose down -v
```

#### 文件结构

```
api-platform/
├── docker/
│   ├── docker-compose.yml       # 编排配置
│   ├── docker-compose.prod.yml  # 生产环境配置
│   ├── frontend/
│   │   ├── Dockerfile           # 前端镜像
│   │   └── nginx.conf           # Nginx配置
│   ├── backend/
│   │   └── Dockerfile           # 后端镜像
│   └── .env.example             # 环境变量示例
└── scripts/
    ├── deploy.sh                # 部署脚本
    ├── backup.sh                # 备份脚本
    └── restore.sh               # 恢复脚本
```

### 方案二: CI/CD 自动化部署

#### 架构图

```
┌─────────────────────────────────────────────────────────────┐
│                    CI/CD 自动化部署架构                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [开发者]                                                   │
│       │                                                     │
│       ▼  git push                                           │
│                                                             │
│   [Git Repository]                                          │
│   (GitHub/GitLab/Gitee)                                    │
│       │                                                     │
│       ▼  webhook                                           │
│                                                             │
│   [CI/CD Server]                                           │
│   (GitHub Actions / GitLab CI / Jenkins)                    │
│       │                                                     │
│       ├──▶ 1. 代码检查 (Lint)                              │
│       ├──▶ 2. 单元测试 (Test)                              │
│       ├──▶ 3. 构建镜像 (Build)                             │
│       ├──▶ 4. 推送镜像 (Push)                              │
│       └──▶ 5. 部署到服务器 (Deploy)                        │
│               │                                             │
│               ▼                                             │
│   [生产服务器]                                              │
│   ┌─────────────────────────────────┐                      │
│   │  Docker Compose / Kubernetes    │                      │
│   │  - 拉取最新镜像                 │                      │
│   │  - 滚动更新服务                 │                      │
│   │  - 健康检查                     │                      │
│   └─────────────────────────────────┘                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 优点

1. **全自动化**: 推送代码自动触发部署
2. **质量保障**: 自动运行测试和代码检查
3. **版本管理**: 每次部署都有Git记录
4. **快速回滚**: 一键回滚到任意历史版本
5. **多环境支持**: 开发、测试、生产环境独立部署
6. **团队协作**: 代码审查 + CI 保障代码质量

#### 缺点

1. **配置复杂**: 需要学习CI/CD工具
2. **调试困难**: 流水线出错时排查较麻烦
3. **依赖网络**: 需要访问代码仓库和镜像仓库
4. **成本较高**: 需要CI/CD服务器或付费服务

#### 实施步骤 (以 GitHub Actions 为例)

```yaml
# .github/workflows/deploy.yml

name: Deploy to Production

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Tests
        run: |
          cd api-platform
          pip install -r requirements.txt
          pytest

  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build Docker Image
        run: |
          docker build -t api-platform:${{ github.sha }} .
          docker tag api-platform:${{ github.sha }} api-platform:latest
      
      - name: Push to Registry
        run: |
          echo ${{ secrets.DOCKER_PASSWORD }} | docker login -u ${{ secrets.DOCKER_USERNAME }} --password-stdin
          docker push api-platform:${{ github.sha }}

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Server
        uses: appleboy/ssh-action@master
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SERVER_SSH_KEY }}
          script: |
            cd /opt/api-platform
            docker-compose pull
            docker-compose up -d
            docker-compose ps
```

#### 文件结构

```
api-platform/
├── .github/
│   └── workflows/
│       ├── ci.yml              # CI流水线 (测试、构建)
│       ├── cd.yml              # CD流水线 (部署)
│       └── release.yml         # 发布流水线
├── .gitlab-ci.yml              # GitLab CI配置 (可选)
└── Jenkinsfile                 # Jenkins配置 (可选)
```

```
│   │  5. 解压程序文件                                │                      │

│   └─────────────────────────────────┘                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 优点

1. **用户体验好**: 双击安装，无需命令行
2. **适合离线**: 可以打包所有依赖
3. **私有化部署**: 适合交付给客户
4. **跨平台**: 可生成 Windows、Mac、Linux 安装包

#### 缺点

1. **包体积大**: 需要打包 Docker Desktop + 镜像
2. **资源占用高**: Electron + Docker 双重开销
3. **更新困难**: 需要重新下载完整安装包
4. **不适合大型生产**: 缺乏专业的运维能力

#### 实施步骤

```bash
# 1. 安装 Node.js (已安装可跳过)
# https://nodejs.org/

# 2. 安装 Electron Builder
cd api-platform/electron
npm install -g electron-builder

# 3. 配置 package.json
{
  "name": "api-platform",
  "version": "1.0.0",
  "scripts": {
    "pack": "electron-builder --dir",
    "dist": "electron-builder"
  },
  "build": {
    "appId": "com.api-platform.app",
    "files": ["**/*", "!**/node_modules/*/{CHANGELOG.md,README.md,README,readme.md,readme}"],
    "win": {
      "target": "nsis",
      "icon": "build/icon.ico"
    },
    "nsis": {
      "oneClick": false,
      "allowToChangeInstallationDirectory": true,
      "createDesktopShortcut": true
    }
  }
}

# 4. 打包
npm run dist

# 5. 生成安装包
# Windows: dist/API Platform Setup 1.0.0.exe
# Mac: dist/API Platform-1.0.0.dmg
# Linux: dist/API Platform_1.0.0_amd64.deb
```

#### 文件结构

```
api-platform/
├── electron/
│   ├── main.js                # Electron 主进程
│   ├── preload.js             # 预加载脚本
│   ├── package.json           # Electron 配置
│   ├── build/                 # 打包资源
│   │   ├── icon.ico
│   │   └── installer.nsh     # NSIS 安装脚本
│   └── docker/                # 内置 Docker 配置
│       └── docker-compose.yml
└── dist/                      # 生成的安装包
    ├── API Platform Setup 1.0.0.exe
    └── API Platform-1.0.0.dmg
```

### 根据团队规模推荐

| 团队规模 | 推荐方案 | 理由 |
|---------|---------|------|
