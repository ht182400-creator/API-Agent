## 问题78：Pytest 测试修复 - 模型关系与字段匹配问题

### 78.1 问题背景

**发现时间**：2026-04-20 14:35

运行 `pytest tests/test_backend_proxy.py -v` 时遇到多个错误，导致测试无法通过。

### 78.2 问题描述

#### 78.2.1 错误1：Payment 模型关系缺失

```
SQLAlchemy: Mapper 'Mapper[User(users)]' has no property 'payments'
```

**原因**：`Payment` 模型使用 `back_populates="payments"` 引用了 `User` 模型不存在的属性。

#### 78.2.2 错误2：UUID 对象序列化失败

```
TypeError: Object of type UUID is not JSON serializable
```

**原因**：JWT token 创建时传入了 UUID 对象，导致 JSON 序列化失败。

#### 78.2.3 错误3：字段不存在

```
TypeError: 'request_timeout' is an invalid keyword argument for Repository
TypeError: 'is_active' is an invalid keyword argument for Repository
```

**原因**：测试中使用了模型中不存在的字段名。

#### 78.2.4 错误4：字段名不匹配

```
AttributeError: 'User' object has no attribute 'access_token'
```

**原因**：测试期望的字段与实际模型结构不一致。

### 78.3 解决方案

#### 78.3.1 修复 User 模型 - 添加 payments 关系

**文件**：`api-platform/src/models/user.py`

```python
# 在 User 类中添加 payments 关系
payments = relationship(
    "Payment",
    back_populates="user",
    cascade="all, delete-orphan",
    lazy="selectin"
)
```

**原因**：`Payment` 模型中定义了 `user = relationship("User", back_populates="payments")`，需要在 `User` 模型中添加对应的反向关系。

#### 78.3.2 修复测试 - UUID 序列化问题

**文件**：`api-platform/tests/conftest.py`

```python
# 修改前
token = create_access_token({"sub": test_user.id})

# 修改后
token = create_access_token({"sub": str(test_user.id)})
```

**原因**：JWT payload 中的 `sub` 字段需要是字符串，而不是 UUID 对象。

#### 78.3.3 修复测试 - 字段名匹配

**Repository 模型字段**：
- 使用 `owner_id` 而非 `created_by`
- 使用 `status` 而非 `is_active`
- 无 `request_timeout` 字段

**API Key 模型字段**：
- 使用 `hash_password()` 处理密码哈希
- `is_active` 在 API Key 模型中存在

### 78.4 修复后的测试结构

#### 78.4.1 test_payment.py

```python
class TestPaymentModel:
    """Payment 模型测试"""
    
    def test_payment_model_structure(self, db_session, test_user):

```

1. **不要简化硬编码测试**：测试必须正确匹配实际模型结构，否则人工测试时仍会遇到相同问题

## 问题80：Recharge.tsx 图标导入错误

**发现时间**：2026-04-20 15:16

```
Uncaught SyntaxError: The requested module '/node_modules/.vite/deps/@ant-design_icons.js?v=8831ce14'
```

```typescript
<span className={styles.priceSection}>
  <YuanOutlined className={styles.currencyIcon} />
  <span className={styles.price}>{pkg.price.toFixed(2)}</span>
</span>
```

**修改后**：
```typescript
import { 
  GiftOutlined, 
  CheckCircleOutlined, 
  WechatOutlined, 
  AlipayOutlined, 
  CreditCardOutlined,
  ReloadOutlined,
  ExclamationCircleOutlined

<span className={styles.priceSection}>
  <span className={styles.currencyIcon}>¥</span>

```

## 问题82：Vite 端口占用导致新人困惑

### 82.1 问题背景

**发现时间**：2026-04-20 18:02

新人不知道端口被占用时 Vite 会自动切换端口，导致困惑和文档不一致。

### 82.2 问题描述

1. 配置文件中设置端口 8080
2. 8080 被占用时 Vite 自动切换到 3000
3. 新人不知道原因，以为配置有问题

### 82.3 解决方案

#### 82.3.1 安装 portfinder 依赖

**文件**：`api-platform/web/package.json`

```json
"devDependencies": {
  "portfinder": "^1.0.32",
  "@playwright/test": "^1.59.1",
  ...
}
```

#### 82.3.2 更新 Vite 配置

**文件**：`api-platform/web/vite.config.ts`

```typescript
import portfinder from 'portfinder'

portfinder.basePort = 3000
portfinder.highestPort = 9999

export default defineConfig(async () => {
  const port = await portfinder.getPortPromise()
  
  return {
    plugins: [react()],
    server: {
      port,
      host: '0.0.0.0',
      onPortWarn: () => {},
      proxy: {
        '/api': { target: 'http://localhost:8000', changeOrigin: true },
      },
    },
  }
})
```

#### 82.3.3 更新后端 CORS 配置

## 问题86：自定义金额充值成功后账单没有显示

### 86.1 问题背景

**发现时间**：2026-04-20 18:49

用户充值1元成功后，账单明细页面没有显示充值记录，账户余额也没有更新。

### 86.2 问题分析

充值成功后调用了回调接口 `POST /api/v1/payments/callback`，但账单仍然没有创建。

调试后发现问题出在 `payment_service.py` 的 `_process_successful_payment` 方法：

```python
# 获取充值套餐
package = await self.get_package(str(payment.package_id))  # package_id 为 None
bonus = float(package.bonus_amount) if package.bonus_amount else 0  # 报错
```

**根本原因**：
- 使用**自定义金额充值**时，`payment.package_id` 为 `None`（没有选择固定套餐）
- 代码直接调用 `await self.get_package(None)` 导致获取套餐信息失败
- 后续创建账单的逻辑没有执行

### 86.3 问题代码

```python
async def _process_successful_payment(self, payment: Payment) -> None:
    # ...
    package = await self.get_package(str(payment.package_id))  # None 转换后仍为 None
    
    # 如果 package_id 为 None，package 为 None，以下代码会报错
    bonus = float(package.bonus_amount) if package.bonus_amount else 0  # AttributeError
```

### 86.4 解决方案

**文件**：`api-platform/src/services/payment_service.py`

修改 `_process_successful_payment` 方法，增加对 `package_id` 为空情况的处理：

```python
async def _process_successful_payment(self, payment: Payment) -> None:
    """
    处理成功支付的逻辑：增加账户余额
    
    Args:
        payment: 支付记录
    """
    from src.services.account_service import AccountService
    from src.config.settings import settings
    
    # 计算到账金额（本金 + 赠送）
    principal = float(payment.amount)  # 本金
    
    # 获取充值套餐信息（如果有）
    package = None
    package_name = "自定义充值"
    bonus = 0.0
    
    if payment.package_id:
        package = await self.get_package(str(payment.package_id))
    
    if package:
        package_name = package.name
        bonus = float(package.bonus_amount) if package.bonus_amount else 0
        
        # 如果有赠送比例
        if package.bonus_ratio and float(package.bonus_ratio) > 0:
            bonus = principal * float(package.bonus_ratio)
    else:
        # 自定义充值，使用全局赠送比例
        if settings.recharge_default_bonus_ratio > 0:
            bonus = principal * settings.recharge_default_bonus_ratio
    
    total_amount = principal + bonus
    
    # 更新账户余额（account_service.add_balance 内部已创建 Bill）
    account_service = AccountService(self.db)
    await account_service.add_balance(
        user_id=str(payment.user_id),
        amount=total_amount,
        source_type="recharge",
        source_id=str(payment.id),
        description=f"充值：{package_name}，金额{principal:.2f}元",
    )
```

### 86.5 关键修改点

1. **添加空值检查**：`if payment.package_id:` 先判断是否为空
2. **自定义充值处理**：使用全局赠送比例 `settings.recharge_default_bonus_ratio`
3. **默认值设置**：套餐名为"自定义充值"

### 86.6 测试步骤

1. 重启后端服务
2. 进入充值中心 → 选择"自定义金额"
3. 输入金额（如1元）→ 选择支付方式 → 支付
4. 确认模拟支付
5. 进入账单中心，验证充值记录已显示

```tsx
} else {
  // 真实支付模式（生产环境）
  message.loading({ content: '获取支付链接...', key: 'payUrl' })
  try {
    const status = await paymentApi.getPaymentStatus(currentPayment.payment_no)
    if (status.pay_url) {
      window.open(status.pay_url, '_blank', 'width=800,height=600')
      message.success({ content: '支付页面已打开，请在页面中完成支付', key: 'payUrl' })

    message.error({ content: '获取支付状态失败', key: 'payUrl' })
  }
}
```

### 102.3 问题2：空值导致的方法调用错误

**错误信息**：
```
TypeError: Cannot read properties of null (reading 'toFixed')
    at render (Reconciliation.tsx:267:46)
```

**原因分析**：
后端API返回的数据中，部分字段（如 `local_amount`、`channel_amount`、`diff_amount`）可能为 `null` 或 `undefined`。直接调用 `.toFixed(2)` 方法会导致运行时错误。

**涉及位置**：
1. 差异记录表格 - 本地金额列
2. 差异记录表格 - 渠道金额列
3. 差异记录表格 - 差异金额列
4. 历史记录表格 - 平台交易金额
5. 历史记录表格 - 渠道交易金额
6. 处理差异弹窗 - 金额显示

**修复方案**：使用空值合并运算符 `??` 或三元检查

**修复示例1 - 表格列渲染**：
```tsx
// 修复前
render: (amount: number) => `¥${amount.toFixed(2)}`

// 修复后
render: (amount: number) => amount != null ? `¥${amount.toFixed(2)}` : '-'
```

**修复示例2 - 历史记录表格**：
```tsx
// 修复前
<Text type="secondary">¥{record.platform_trade_amount.toFixed(2)}</Text>

// 修复后
<Text type="secondary">¥{(record.platform_trade_amount ?? 0).toFixed(2)}</Text>
```

**修复示例3 - 弹窗金额显示**：

```python
# src/services/reconciliation_scheduler.py
from apscheduler.schedulers.asyncio import AsyncIOScheduler
from apscheduler.triggers.cron import CronTrigger

class ReconciliationScheduler:
    def __init__(self):
        self._scheduler = AsyncIOScheduler()
        self._is_running = False
        self._last_run_time = None
        self._tasks = []
    
    async def execute_reconciliation(
        self,
        date: Optional[str] = None,
        channels: Optional[List[str]] = None
    ) -> Dict[str, Any]:
        """执行对账任务，默认T+1"""
        if date is None:
            date = (datetime.now() - timedelta(days=1)).strftime('%Y-%m-%d')
        
        # 调用对账服务执行对账
        result = await reconciliation_service.execute_reconciliation(
            date=date,
            channels=channels or ['alipay', 'wechat', 'bankcard']
        )
        self._last_run_time = datetime.now()
        return result
    
    def start(self):
        """启动调度器"""
        # 添加每日凌晨2点执行的任务
        trigger = CronTrigger(hour=2, minute=0)
        self._scheduler.add_job(
            self.execute_reconciliation,
            trigger=trigger,
            id='daily_reconciliation',
            name='每日对账任务'
        )
        self._scheduler.start()
        self._is_running = True
    
    def get_task_status(self) -> Dict[str, Any]:
        """获取定时任务状态"""
        return {
            "is_running": self._is_running,
            "last_run_time": self._last_run_time.isoformat() if self._last_run_time else None,
            "task_count": len(self._tasks),
            "tasks": self._tasks,
        }

# 全局调度器实例
scheduler = ReconciliationScheduler()
```

## 问题106：账单中心调用次数与消费金额对应关系分析

### 106.1 问题描述

用户反馈：开发者的账单中心里面"本月调用次数"和"本月消费金额"是如何对应的？这个消费比例是否可以配置？

### 106.2 当前系统分析

#### 106.2.1 相关数据表

**Bill表（账单表）**：
```python
class Bill(Base):
    bill_type: str      # recharge(充值)/consume(消费)/refund(退款)
    amount: str         # 变动金额
    source_type: str    # api_call/manual/refund
    source_id: str      # 关联的API Key ID
```

**APICallLog表（API调用日志）**：
```python
class APICallLog(Base):
    tokens_used: BigInteger  # Token使用量
    cost: str                # 消费金额
```

#### 106.2.2 月度汇总API

当前 `/monthly-summary` 返回：
```json
{
  "total_recharge": 100.00,
  "total_consumption": 50.00,
  "consumption_count": 25,  // ❌ 这是账单记录数，不是API调用次数
  "net_change": 50.00
}
```

#### 106.2.3 当前问题

| 问题 | 说明 |
|------|------|
| `consumption_count` | 是Bill表中`bill_type='consumption'`的记录数 |
| 非真实调用次数 | 不等于实际的API调用次数 |
| 缺少Token统计 | `APICallLog.tokens_used`未被用于月度汇总 |
| 缺少计费配置 | 无法按仓库/接口配置不同费率 |

### 106.3 主流API平台计费模式

| 平台 | 计费方式 | 说明 |
|------|----------|------|
| **OpenAI** | 按Token | 输入Token + 输出Token，不同模型价格不同 |
| **阿里云百炼** | 按Token 或 按调用次数 | 两种模式可选 |
| **AWS API Gateway** | 按调用次数 + 数据传输量 | 请求数 + GB计费 |
| **腾讯云大模型** | 按Token | 输入/输出分开计价 |
| **Azure OpenAI** | 按Token | 与OpenAI类似 |

### 106.4 计费模式对比

| 计费模式 | 优点 | 缺点 | 适用场景 |
|----------|------|------|----------|
| 按调用次数 | 简单直观 | 无法区分调用复杂度 | 简单API |
| 按Token | 精确计费 | 需要Token统计 | 大模型API |
| 混合模式 | 灵活 | 配置复杂 | 综合平台 |

### 106.5 优化建议

#### 方案一：简单按调用次数计费（当前可用）

适用于简单的HTTP API服务。

```
消费金额 = 调用次数 × 单次费率
```

**优点**：实现简单，用户容易理解
**缺点**：无法区分调用复杂度

#### 方案二：按Token计费（推荐）

适用于AI/大模型类API服务。

```
消费金额 = (输入Token数 × 输入单价 + 输出Token数 × 输出单价)
```

**优点**：精确计费，公平合理
**缺点**：需要准确统计Token

#### 方案三：混合计费（高级）

支持按仓库配置不同计费模式。

```python
# 仓库级别计费配置
{
  "repo_id": "uuid",
  "pricing_type": "token",  # call/token/package
  "price_per_call": 0.1,   # 单次调用价格
  "price_per_1k_tokens": 0.1,  # 每千Token价格
}
```

#### 107.3.2 费用计算方法

```python
def calculate_cost(
    self,
    call_count: int = 1,
    input_tokens: int = 0,
    output_tokens: int = 0,
    vip_level: int = 0
) -> float:
    """计算费用"""
    if not self.is_valid():
        return 0.0

    total_cost = 0.0

    if self.pricing_type == "call":
        billable_calls = max(0, call_count - self.free_calls)
        unit_price = float(self.price_per_call or 0)
        total_cost = billable_calls * unit_price

    elif self.pricing_type == "token":
        billable_input = max(0, input_tokens - self.free_input_tokens)
        billable_output = max(0, output_tokens - self.free_output_tokens)
        input_price = float(self.price_per_1k_input_tokens or 0) / 1000
        output_price = float(self.price_per_1k_output_tokens or 0) / 1000
        total_cost = billable_input * input_price + billable_output * output_price

    elif self.pricing_type == "package":
        # 套餐包计费在扣费时单独处理
        total_cost = 0.0

    # 应用阶梯折扣
    tier_discount = self.get_tier_discount(call_count)
    total_cost *= tier_discount

    # 应用VIP折扣
    vip_discount = self.get_vip_discount(vip_level)
    total_cost *= vip_discount

    return round(total_cost, 4)
```

### 107.4 遇到的问题及解决

#### 问题1：PostgreSQL 不支持多条语句一起执行

**错误信息**：
```
asyncpg.exceptions.PostgresSyntaxError: 无法插入多条命令到一个准备好的语句中
```

**原因**：asyncpg 不支持在单个 `execute()` 调用中执行多条 SQL 语句。

**解决方案**：将每条 SQL 语句拆分为独立的 `await conn.execute(text(...))` 调用。

```python
# 错误写法
await conn.execute(text("""
    CREATE INDEX IF NOT EXISTS idx_1 ON table(col1);
    CREATE INDEX IF NOT EXISTS idx_2 ON table(col2);
"""))

# 正确写法
await conn.execute(text("CREATE INDEX IF NOT EXISTS idx_1 ON table(col1)"))
await conn.execute(text("CREATE INDEX IF NOT EXISTS idx_2 ON table(col2)"))
```

#### 问题2：触发器和函数需要分开执行

**错误信息**：同样的 "无法插入多条命令" 错误

**解决方案**：将函数创建和触发器创建分开执行

```python
# 创建函数
await conn.execute(text("""
    CREATE OR REPLACE FUNCTION update_updated_at()
    RETURNS TRIGGER AS $$
    BEGIN
        NEW.updated_at = CURRENT_TIMESTAMP;
        RETURN NEW;
    END;
    $$ language 'plpgsql'
"""))

# 删除旧触发器
await conn.execute(text("DROP TRIGGER IF EXISTS trigger_name ON table_name"))

# 创建新触发器
await conn.execute(text("CREATE TRIGGER trigger_name BEFORE UPDATE ON table_name FOR EACH ROW EXECUTE FUNCTION update_updated_at()"))
```

#### 109.4.2 P3：调用时实时计费

**使用者**：系统内部逻辑（不直接面向用户）

**功能描述**：
这是 API Gateway / Middleware 层的底层逻辑，在每次 API 调用时自动计算费用：
1. 用户发起 API 请求
2. 系统根据请求的仓库获取对应的计费配置
3. 根据配置计算本次调用的费用
4. 记录调用日志（包含费用）
5. 异步扣费

**不暴露给普通用户的原因**：
- 计费逻辑复杂，用户不需要知道细节
- 费用应该在结算时展示，而不是每次调用时
- 避免用户因担心费用而限制正常使用

**Developer 是否需要？**
不需要。Developer 只需要：
- 查看自己的余额
- 查看自己的消费明细

### 113.5 问题3：SQL select 语句括号不匹配

#### 113.5.1 错误信息

```
SyntaxError: unmatched ')'
```

发生在 `admin_billing.py` 第 787 行。

#### 113.5.2 原因分析

`select()` 函数调用中，括号嵌套层级错误：

**错误代码**：
```python
consume_result = await db.execute(
    select(func.coalesce(func.sum(func.abs(func.cast(Bill.amount, Decimal)))), 0)).where(
        Bill.user_id == uid,
        # ...
    )
)
```

问题在于 `select(...)` 末尾有两个 `))`，但实际只需要一个。`select(func.coalesce(...)), 0)` 多了一个右括号。

#### 113.5.3 解决方案

**文件**：`api-platform/src/api/v1/admin_billing.py`

**修改前**（第779-788行）：
```python
# 查询本月消费总额
consume_result = await db.execute(
    select(func.coalesce(func.sum(func.abs(func.cast(Bill.amount, Decimal)))), 0)).where(
        Bill.user_id == uid,
        Bill.bill_type == "consumption",
        Bill.environment == environment,
        Bill.created_at >= start_date,
        Bill.created_at < end_date,
    )
)
```

**修改后**：
```python
# 查询本月消费总额
consume_result = await db.execute(
    select(func.coalesce(func.sum(func.abs(func.cast(Bill.amount, Decimal))), 0)).where(
        Bill.user_id == uid,
        Bill.bill_type == "consumption",
        Bill.environment == environment,
        Bill.created_at >= start_date,
        Bill.created_at < end_date,
    )
)
```

**关键修改**：`)), 0)` → `))), 0)`（在 `Decimal))` 后加一个 `)` 闭合 `coalesce`）

#### 113.5.4 括号匹配分析

```python
select(                              # 1. select 开始
    func.coalesce(                   # 2. coalesce 开始
        func.sum(                    # 3. sum 开始
            func.abs(                # 4. abs 开始
                func.cast(           # 5. cast 开始
                    Bill.amount,      # 参数
                    Decimal           # 参数
                )                    # 5. cast 结束
            )                        # 4. abs 结束
        ),                           # 3. sum 结束
        0                            # 2. coalesce 第二个参数
    )                                # 2. coalesce 结束  ← 关键：这里应该只有一个右括号
)                                    # 1. select 结束
```

## 114. 费用预览 API 失败与月度账单 404 问题排查

### 114.1 问题描述

**问题1**：前端费用预览功能显示"费用计算失败"

**问题2**：前端访问 `/admin/billing/monthly-bills/generated` 返回 404 Not Found

### 114.2 排查过程

#### 问题1：费用计算失败

**错误日志**：
```
400 "无效的配置ID格式"
```

**排查步骤**：

1. **检查前端 API 调用**
   - 文件：`web/src/api/adminPricingConfig.ts`
   - 前端使用 `GET` 方法，参数通过 Query 传递

2. **检查后端路由定义**
   - 文件：`src/api/v1/admin_pricing_config.py`
   - 原始代码使用 `@router.post("/calculate-cost")`

3. **发现根因：FastAPI 路由顺序问题**
   - `/calculate-cost` 具体路径放在了 `/{config_id}` 参数化路径之后
   - FastAPI 将 `calculate-cost` 当作 `config_id` 参数值处理，导致 UUID 验证失败

4. **修复方案**：
   - 将 `/calculate-cost` 路由移到 `/{config_id}` 之前
   - 将 HTTP 方法从 POST 改为 GET（匹配前端）

#### 问题2：月度账单 404

**错误日志**：
```
GET /admin/billing/monthly-bills/generated
404 Not Found
```

**排查步骤**：

1. **检查路由定义**
   - 文件：`src/api/v1/admin_billing.py`
   - 路由已正确定义：`@router.get("/monthly-bills/generated")`

2. **检查路由注册**
   - 文件：`src/api/v1/__init__.py`
   - 发现问题：`admin_billing_router` 注册时没有设置 `prefix`

3. **发现根因：路由注册缺失 prefix**
   - 实际路径：`/monthly-bills/generated`
   - 前端请求：`/admin/billing/monthly-bills/generated`
   - 路径不匹配导致 404

4. **修复方案**：
   ```python
   # 修复前
   api_router.include_router(admin_billing_router, tags=["Admin Billing"])
   
   # 修复后
   api_router.include_router(admin_billing_router, prefix="/admin/billing", tags=["Admin Billing"])
   ```

### 114.3 涉及文件修改

| 文件 | 修改内容 |
|------|----------|
| `web/src/api/adminPricingConfig.ts` | 修改 `calculateCost` 使用 GET + Query 参数 |
| `src/api/v1/admin_pricing_config.py` | 路由改为 GET，调整路由顺序，移除无效的 UUID 比较 |
| `src/api/v1/__init__.py` | 添加 `prefix="/admin/billing"` 到 `admin_billing_router` 注册 |

### 114.4 关键经验总结

#### 1. FastAPI 路由顺序原则

**重要**：FastAPI 中，具体路径路由必须放在参数化路由之前。

```python
# ✅ 正确顺序
@router.get("/calculate-cost")  # 具体路径在前
@router.get("/{config_id}")      # 参数化路由在后

# ❌ 错误顺序（会导致 /calculate-cost 被当作 config_id 参数）
@router.get("/{config_id}")      # 参数化路由在前
@router.get("/calculate-cost")  # 具体路径在后
```

#### 2. APIRouter 注册必须指定 prefix

确保所有子路由注册时有正确的 prefix：

```python
# ✅ 正确示例
api_router.include_router(admin_router, prefix="/admin", tags=["Admin"])
api_router.include_router(billing_router, prefix="/admin/billing", tags=["Admin Billing"])

# ❌ 错误示例（缺失 prefix）
api_router.include_router(billing_router, tags=["Admin Billing"])
```

#### 3. 前后端 HTTP 方法一致性

- 前端使用 GET → 后端必须使用 `@router.get`
- 前端使用 POST → 后端必须使用 `@router.post`
- 混用会导致 405 Method Not Allowed 错误

#### 4. Query 参数 vs Body 参数

- 前端 `axios.get(url, { params: {...} })` → 后端使用 `Query(...)`
- 前端 `axios.post(url, data)` → 后端使用 `Body(...)` 或 Pydantic Model

### 114.5 排查清单

遇到类似问题时，按以下顺序排查：

1. **检查 HTTP 方法**：前端 GET/POST 与后端 `@router.get/post` 是否一致
2. **检查参数传递方式**：Query 参数 vs Body 参数
3. **检查路由顺序**：具体路径是否在参数化路径之前
4. **检查路由注册**：prefix 是否正确设置
5. **检查服务是否重启**：修改后端代码需要重启服务

### 114.6 预防措施

1. **统一命名规范**：路由路径使用 kebab-case（如 `/monthly-bills`）
2. **路由注册检查表**：新增 router 时务必检查 prefix
3. **前后端 API 契约**：使用 Swagger/OpenAPI 文档确保一致
4. **自动化测试**：编写 API 集成测试验证路由正确性
