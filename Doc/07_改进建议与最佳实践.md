#### 7.2.3 导出功能实现

导出功能必须使用 `fetch` + Blob 方式实现，以支持携带认证 Header：

```typescript
/**
 * 导出账单为 CSV 文件
 * @param params 导出参数（可选）
 */
exportBills: async (params?: {
  bill_type?: string
  start_date?: string
  end_date?: string
}) => {
  try {
    // 获取认证 Token
    const token = useAuthStore.getState().accessToken
    
    // 构建查询参数
    const queryParams = new URLSearchParams()
    if (params?.bill_type) queryParams.append('bill_type', params.bill_type)
    if (params?.start_date) queryParams.append('start_date', params.start_date)
    if (params?.end_date) queryParams.append('end_date', params.end_date)
    
    const queryString = queryParams.toString() ? `?${queryParams.toString()}` : ''
    const baseUrl = import.meta.env.VITE_API_URL || '/api/v1'
    
    // 使用 fetch 发送带认证的请求
    const response = await fetch(`${baseUrl}/billing/bills/export${queryString}`, {
      method: 'GET',
      headers: {
        'Authorization': `Bearer ${token}`,
      },
    })
    
    // 检查响应状态
    if (!response.ok) {
      throw new Error(`导出失败 (${response.status})`)
    }
    
    // 下载文件
    const blob = await response.blob()
    const url = window.URL.createObjectURL(blob)
    const link = document.createElement('a')
    link.href = url
    link.download = `bills_${Date.now()}.csv`
    link.click()
    window.URL.revokeObjectURL(url)
  } catch (error) {
    // 记录详细错误日志
    console.error('[导出账单] 失败:', error)
    // 向用户显示友好错误信息
    throw new Error('导出失败，请稍后重试')
  }
}
```

**注意**：禁止使用 `window.open()` 实现导出功能，因为该方法无法携带 `Authorization` Header。

## 第九部分：数据库关联管理

### 9.1 SQLAlchemy 级联删除配置

#### 9.1.1 问题描述

在多表关联场景中，当删除父记录时，子表关联数据若未配置级联删除，会导致：

1. **数据孤岛**：孤儿记录占用数据库空间
2. **外键约束失败**：部分数据库严格模式会报错
3. **业务逻辑不一致**：残留数据可能影响统计和报表

#### 9.1.2 问题场景

本项目中的 Repository 仓库与其他业务表存在多对一关系：

| 关联表 | 关联字段 | 说明 |
|--------|----------|------|
| `repo_endpoints` | repo_id | API端点配置 |
| `repo_limits` | repo_id | 限流配置 |
| `repo_pricing` | repo_id | 定价配置 |
| `repo_configs` | repo_id | 仓库配置 |
| `repo_stats` | repo_id | 统计记录 |

#### 9.1.3 解决方案

在 SQLAlchemy 模型中为 relationship 添加 `cascade="all, delete-orphan"` 配置：

```python
# Repository 模型中的关系配置
class Repository(Base):
    __tablename__ = "repositories"

    # ... 其他字段 ...

    # Relationships - 添加 cascade 配置
    pricing = relationship("RepoPricing", back_populates="repository",
                           uselist=False, cascade="all, delete-orphan")
    pricing_configs = relationship("PricingConfig", back_populates="repository",
                                   cascade="all, delete-orphan")
    configs = relationship("RepoConfig", back_populates="repository",
                           cascade="all, delete-orphan")
    endpoints = relationship("RepoEndpoint", back_populates="repository",
                              order_by="RepoEndpoint.display_order",
                              cascade="all, delete-orphan")
    limits = relationship("RepoLimits", back_populates="repository",
                          uselist=False, cascade="all, delete-orphan")
    stats = relationship("RepoStats", back_populates="repository",
                         cascade="all, delete-orphan")
```

#### 9.1.4 cascade 参数说明

| 参数值 | 说明 |
|--------|------|
| `all` | 等同于 `save-update, merge, delete, delete-orphan` |
| `delete-orphan` | 当对象从父对象的列表中移除时，自动删除子对象 |
| `save-update` | 当对象添加到 session 时，自动保存关联对象 |
| `merge` | 合并操作时合并关联对象 |

#### 9.1.5 影响分析

| 影响维度 | 影响程度 | 说明 |
|----------|----------|------|
| 数据库表结构 | **无影响** | 不需要修改表字段 |
| API 接口 | **无影响** | 不需要修改任何接口 |
| 前端代码 | **无影响** | 不需要修改任何页面 |
| 业务逻辑 | **正向影响** | 删除时数据自动清理，更安全 |

#### 9.1.6 验证方法

```python
# 验证级联删除配置
from src.models.repository import Repository

repo = db.query(Repository).filter(Repository.id == repo_id).first()

# 查看 relationship 配置
for rel_name, rel in Repository.__mapper__.relationships.items():
    print(f"{rel_name}: cascade={rel.cascade}")
```

#### 9.1.7 注意事项

1. **测试环境验证**：生产部署前在测试环境验证级联删除效果
2. **备份策略**：删除操作前确保有数据备份
3. **软删除优先**：对于重要数据可考虑软删除（添加 `deleted_at` 字段）
4. **外键约束**：确保子表外键已配置 `ondelete="CASCADE"`（数据库层面）

## 第十二部分：用户操作日志记录规范

### 12.1 概述

用户操作日志记录是系统可观测性和安全审计的重要组成部分。通过记录用户在界面上的关键操作，可以实现：
- 问题追踪：用户操作问题快速定位
- 安全审计：敏感操作留痕可查
- 行为分析：用户使用习惯分析
- 合规要求：满足监管审计需求

### 12.2 记录原则

| 原则 | 说明 | 实践 |
|------|------|------|
| 必要性 | 只记录关键操作 | 避免过度记录 |
| 最小化 | 只记录必要字段 | 避免敏感信息泄露 |
| 异步性 | 日志记录不影响主流程 | 使用异步写入 |
| 可追溯 | 操作可关联到具体用户 | 用户ID必填 |

### 12.3 记录操作类型

| 操作类型 | 操作类型值 | 说明 | 优先级 |
|----------|------------|------|--------|
| 用户登录 | login | 登录成功 | P0 |
| 用户登出 | logout | 退出登录 | P1 |
| 升级为开发者 | upgrade_to_developer | 点击升级按钮 | P0 |
| 领取试用金额 | claim_trial | 点击领取试用 | P0 |
| 账户充值 | recharge | 充值操作 | P0 |
| 创建API Key | create_api_key | 创建密钥 | P1 |
| 删除API Key | delete_api_key | 删除密钥 | P1 |
| 创建仓库 | create_repo | 创建仓库 | P1 |
| 删除仓库 | delete_repo | 删除仓库 | P1 |
| 结算提现 | settlement | 收益结算 | P0 |

```python
# src/utils/device_detector.py
from typing import Dict, Optional

def detect_device(user_agent: str) -> Dict[str, Optional[str]]:
    """
    从 User-Agent 解析设备信息
    
    Returns:
        {
            "device_type": "desktop/mobile/tablet",
            "browser": "Chrome/Firefox/Safari/...",
            "os": "Windows/macOS/Linux/iOS/Android"
        }
    """
    ua = user_agent.lower()
    
    # 设备类型识别
    if any(mobile in ua for mobile in ['iphone', 'android', 'mobile']):
        device_type = "mobile"
    elif any(tablet in ua for tablet in ['ipad', 'tablet']):
        device_type = "tablet"
    else:
        device_type = "desktop"
    
    # 浏览器识别
    if 'chrome' in ua and 'edg' not in ua:
        browser = "Chrome"
    elif 'firefox' in ua:
        browser = "Firefox"
    elif 'safari' in ua and 'chrome' not in ua:
        browser = "Safari"
    elif 'edg' in ua:
        browser = "Edge"
    else:
        browser = "Other"
    
    # 操作系统识别
    if 'windows' in ua:
        os = "Windows"
    elif 'mac os' in ua or 'macos' in ua:
        os = "macOS"
    elif 'linux' in ua:
        os = "Linux"
    elif 'iphone' in ua or 'ipad' in ua:
        os = "iOS"
    elif 'android' in ua:
        os = "Android"
    else:
        os = "Other"
    
    return {
        "device_type": device_type,
        "browser": browser,
        "os": os
    }
```

## 第十六部分：用户角色体系设计最佳实践 (V4.0)

### 16.1 问题背景

**发现时间**：2026-04-23

在 V4.0 重构之前，用户角色体系存在以下问题：

1. **双重角色字段**：`role` 和 `user_type` 两个字段语义不清
2. **owner 角色冗余**：owner 和 developer 功能重叠
3. **权限检查分散**：硬编码检查散落在各处
4. **数据不一致**：部分用户两个字段值不匹配

### 16.2 解决方案

**统一角色体系**：

| 设计原则 | 说明 |
|----------|------|
| 统一标识 | user_type 作为主要角色标识 |
| 向后兼容 | role 字段保留用于兼容旧代码 |
| 业务状态 | owner = developer + 有仓库 |
| 集中检查 | PermissionService 统一权限检查 |

**角色定义**：

| user_type | 说明 | 权限 |
|-----------|------|------|
| user | 普通用户 | 充值、调用API |
| developer | API开发者 | 充值、调用API、创建仓库、获得收益 |
| admin | 管理员 | 平台运营管理 |
| super_admin | 超级管理员 | 最高权限 |

### 16.3 PermissionService 实现

```python
class PermissionService:
    """统一权限检查服务"""

    @staticmethod
    def get_user_role(user: User) -> str:
        """获取统一角色（优先 user_type）"""
        if user.user_type in [r.value for r in UserRole]:
            return user.user_type
        return user.role or UserRole.USER.value

    @staticmethod
    def is_developer(user: User) -> bool:
        """检查是否为开发者"""
        role = PermissionService.get_user_role(user)
        return role in ["developer", "admin", "super_admin"]

    @staticmethod
    def can_create_repo(user: User) -> bool:
        """检查是否可以创建仓库"""
        return PermissionService.is_developer(user)
```

## 第17章：用户升级方案设计 (V4.1)

### 17.1 背景问题

**问题发现**：2026-04-23

当前用户升级流程存在设计缺陷：
- 普通用户点击"升级用户"按钮 → 直接升级为开发者
- 升级操作太轻率，用户可能误操作
- 没有明确告知用户升级的代价和权益

### 17.2 四种升级方案对比

| 方案 | 说明 | 适用场景 | 优先级 |
|------|------|----------|--------|
| **充值升级** | 必须先付费才能升级 | 大多数API平台 | 参考 |
| **付费1元升级** | 象征性收费作为门槛 | 当前需实现 | **P0** |
| **任务升级** | 完成认证/绑定后才可升级 | 需要资质审核 | 参考 |
| **加强提示** | 增加二次确认机制 | 作为辅助手段 | P1 |

### 17.3 方案1：充值升级（参考方案）

**模式**：
```
升级为开发者 → 选择套餐 → 支付 → 升级成功
```

**优点**：
- 用户有明确的付费预期
- 主流SaaS平台常用做法（Stripe、OpenAI等）
- 收益直接转化

**缺点**：
- 付费门槛较高，可能流失用户
- 需要完整的支付流程

**适用场景**：商业化成熟的API平台

---

### 17.4 方案2：付费1元升级（当前实现）⭐

#### 17.4.1 设计思路

**核心原则**：象征性收费，设置最低门槛

```
升级为开发者 → 支付1元 → 立即升级
```

**为什么选择1元**：
- 足够低，不构成付费障碍
- 足够高，用户需要思考决定
- 排除误操作和恶意注册

```python
):
    """

    用户支付1元后自动升级为开发者
    """
    # 1. 检查当前用户是否是普通用户
```

### 17.5 方案3：任务升级（参考方案）

**模式**：
```
升级为开发者 → 完成实名认证/手机绑定 → 免费升级
```

**优点**：
- 免费门槛，用户接受度高
- 收集用户信息，便于运营
- 筛选真实用户

**缺点**：
- 用户可能不愿意提供信息
- 需要对接实名认证服务

**适用场景**：需要严格用户身份的B端平台

**实现思路**：
```python
@router.post("/upgrade-via-task", response_model=BaseResponse[UpgradeResponse])
async def upgrade_via_task(
    current_user: User = Depends(get_current_user),
    db: AsyncSession = Depends(get_db),
):
    """任务升级"""
    # 检查是否已完成必要任务
    if not current_user.phone:

        raise ValidationError("请先完成实名认证")
    
    # 升级用户
    current_user.role = "developer"
    current_user.user_type = "developer"
    
    await db.commit()
    
    return BaseResponse(

        message="升级成功！",
```
