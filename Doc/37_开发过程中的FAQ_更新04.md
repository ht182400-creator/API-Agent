## 问题127：账单中心和其他页面出现"Multiple rows were found"错误

### 127.1 问题描述

**发现时间**：2026-04-23

**影响范围**：
- 普通用户工作台 - 账单中心
- 开发者工作台 - 账户信息
- 充值功能 - 账户查询

**错误信息**：
```
Multiple rows were found when one or none was required
```

**触发场景**：
1. 注册新用户后登录，进入工作台
2. 点击"账单中心"菜单
3. 页面显示500错误

### 127.2 问题原因分析

#### 根本原因：数据库存在重复记录

数据库 `accounts` 表中，同一个 `user_id` 对应多条 `balance` 类型的账户记录。这可能是由于：
1. 历史代码缺陷导致重复创建账户
2. 并发请求导致重复插入
3. 测试数据残留

#### 技术分析

代码中使用 `scalar_one_or_none()` 查询账户：
```python
result = await db.execute(
    select(Account).where(
        Account.user_id == current_user.id,
        Account.account_type == "balance"
    )
)
account = result.scalar_one_or_none()  # 当存在多条记录时抛出异常
```

### 127.3 解决方案

#### 统一修复模式

将所有 `scalar_one_or_none()` 替换为安全的多记录检查：

```python
# 修复前
account = result.scalar_one_or_none()

# 修复后
accounts = result.scalars().all()
if len(accounts) > 1:
    logger.warning(f"用户 {user_id} 存在多个 balance 账户，取第一条")
    account = accounts[0]
elif len(accounts) == 0:
    account = None
else:
    account = accounts[0]
```

   ```sql
   ON accounts(user_id, account_type) 
   WHERE account_type = 'balance';
   ```

2. **代码层面**：在创建账户前先检查是否存在

   ```python
   existing = await db.execute(
       select(Account).where(
           Account.user_id == user_id,

   )
   if existing.scalar_one_or_none():
       raise ValidationError("账户已存在")
   ```

3. **数据清理**：定期检查并清理重复记录
   ```sql
   -- 查找重复记录
   SELECT user_id, account_type, COUNT(*) 
   FROM accounts 
   GROUP BY user_id, account_type 
   HAVING COUNT(*) > 1;
   
   -- 保留最新记录，删除旧记录
   DELETE FROM accounts 
   WHERE id NOT IN (
       SELECT MAX(id) 
       FROM accounts 
       GROUP BY user_id, account_type
   );
   ```

## 问题128：充值成功但余额显示为0

### 128.1 问题描述

**发现时间**：2026-04-23

**现象**：
1. 新注册普通用户充值1元
2. 充值成功，页面跳转到开发者界面
3. 但账户余额显示为0
4. 账单中心余额也是0

**根本原因**：

`_get_account_or_create` 函数（用于体验金领取）与 `account_service.get_or_create_account` 查询条件不一致：

| 函数 | 查询条件 | 问题 |
|------|----------|------|
| `_get_account_or_create` | `user_id == user.id` | 没有过滤 `account_type` |
| `get_or_create_account` | `user_id == user_id AND account_type == "balance"` | 正确过滤 |

**后果**：
1. 体验金领取时：创建了一个 `account_type="balance"` 的账户，余额为10元
2. 充值1元时：使用 `account_service.get_or_create_account`，只查询 `account_type=="balance"` 的账户
3. 如果体验金创建的账户类型不是 `"balance"`（或存在多条记录），充值会创建或更新**不同的账户**
4. 结果：充值金额和体验金分别在不同账户，导致余额显示为0

### 128.2 解决方案

统一 `_get_account_or_create` 函数的查询条件，与 `account_service` 保持一致：

**修改文件**：`src/api/v1/user.py`

**修复前**：
```python
async def _get_account_or_create(db: AsyncSession, user: User) -> Account:
    """获取或创建用户账户"""
    result = await db.execute(
        select(Account).where(Account.user_id == user.id)  # 缺少 account_type 过滤
    )
```

**修复后**：
```python
async def _get_account_or_create(db: AsyncSession, user: User) -> Account:
    """
    获取或创建用户账户
    
    注意：与 account_service.get_or_create_account 保持一致，使用 account_type="balance"
    """
    from sqlalchemy import and_
    
    result = await db.execute(
        select(Account).where(
            and_(
                Account.user_id == user.id,
                Account.account_type == "balance"
            )
        )
    )
```

### 128.3 预防措施

1. **统一账户查询逻辑**：所有涉及账户查询的代码都应使用相同的查询条件
2. **添加数据库约束**：为 `(user_id, account_type)` 添加唯一索引
3. **代码审查**：检查所有账户相关代码，确保查询条件一致

## 问题129：充值成功后余额仍为0（续）- get_account 接口未提交事务

### 129.1 问题描述

**发现时间**：2026-04-23

**现象**：
1. 新注册用户充值1元
2. 充值成功回调执行完成
3. 但工作台和账单中心余额仍显示为0

**根本原因**：

在 `src/api/v1/billing.py` 的 `get_account` 接口中，如果账户不存在，会创建一个新账户，但**只执行了 `flush()` 没有 `commit()`**：

```python

    account = Account(...)
    db.add(account)
    await db.flush()  # ❌ 只 flush，没有 commit！
    # 导致返回的账户数据可能与数据库不一致
```

### 129.2 解决方案

在创建账户后执行 `commit()` 确保数据持久化：

**修改文件**：`src/api/v1/billing.py` 第52-66行

**修复前**：
```python
if not account:
    account = Account(...)
    db.add(account)
    await db.flush()  # 缺少 commit
```

**修复后**：
```python
if not account:
    account = Account(...)
    db.add(account)
    await db.commit()  # ✅ 需要 commit 才能持久化
    await db.refresh(account)
    logger.info(f"[Billing] Created new account for user {current_user.id}: balance={account.balance}")
```

## 问题130：并发场景下创建重复账户（方案A：唯一约束）

### 130.1 问题描述

**发现时间**：2026-04-23

**现象**：
1. 用户注册后访问页面
2. 系统创建了多个账户（Found 2 accounts）
3. 导致充值时操作的不是同一个账户

**根本原因**：
并发请求场景下，多个代码路径同时检查"账户是否存在"，都发现不存在，然后都创建账户。

**涉及的代码路径分析**：

| 序号 | 文件 | 方法 | 用途 | 调用场景 | 必要性 |
|------|------|------|------|----------|--------|
| 1 | `account_service.py` | `get_or_create_account` | 充值/扣费核心操作 | 充值回调、消费扣费 | ✅ 必要 |
| 2 | `user.py` | `_get_account_or_create` | 用户状态检查、试用金额 | 体验金领取、用户状态接口 | ✅ 必要 |
| 3 | `billing.py` | `get_account` | 获取账户信息 | 账单中心、工作台 | ✅ 必要 |
| 4 | `billing_service.py` | `get_or_create_account` | 内部服务 | 死代码，未被调用 | ❌ 可删除 |

**问题根源**：
4个不同的代码路径都在做"检查-创建"的模式，在并发场景下会产生竞争条件。

### 130.2 解决方案对比

针对并发创建重复账户问题，分析了以下3种解决方案：

#### 方案A：数据库唯一约束 + 异常捕获（推荐）

**原理**：在数据库层面保证 `(user_id, account_type)` 组合唯一，代码捕获唯一约束违反异常。

**优点**：
- 数据库层面保证一致性，线程安全
- 代码逻辑简洁，无需复杂的锁机制
- 业界最佳实践
- 性能开销最小

**缺点**：
- 需要清理已有重复数据
- 需要迁移数据库

**实现代码**：
```python
from sqlalchemy.exc import IntegrityError

async def get_or_create_account(self, user_id: str) -> Account:
    # 查询账户
    result = await self.db.execute(
        select(Account).where(
            and_(
                Account.user_id == uuid.UUID(user_id),
                Account.account_type == "balance"
            )
        )
    )
    account = result.scalars().first()
    
    if not account:
        # 创建账户
        account = Account(...)
        self.db.add(account)
        try:
            await self.db.commit()
            await self.db.refresh(account)
        except IntegrityError:
            # 唯一约束冲突：回滚并重新查询
            await self.db.rollback()
            result = await self.db.execute(
                select(Account).where(
                    and_(
                        Account.user_id == uuid.UUID(user_id),
                        Account.account_type == "balance"
                    )
                )
            )
            account = result.scalars().first()
    
    return account
```

#### 方案B：分布式锁（Redis/SQLAlchemy）

**原理**：使用分布式锁确保同一时刻只有一个请求能执行创建操作。

**优点**：
- 实现简单
- 可视化好

**缺点**：
- 引入额外依赖（Redis）
- 增加系统复杂度
- 锁获取失败需要重试
- 性能开销较大

**实现代码**：
```python
# 需要引入 Redis 或数据库锁
async with redis.lock(f"account:{user_id}", timeout=10):
    # 检查并创建账户
    ...
```

#### 方案C：检查-插入原子化

**原理**：使用 `INSERT ... ON CONFLICT DO NOTHING` 语法。

**优点**：
- 单条SQL完成
- 无需异常处理

**缺点**：
- SQLAlchemy 语法较复杂
- 不同数据库语法不同

**实现代码**：
```python
from sqlalchemy.dialects.postgresql import insert

stmt = insert(Account).values(...).on_conflict_do_nothing(
    index_elements=['user_id', 'account_type']
)
await self.db.execute(stmt)
```

#### 最终选择

**选择方案A**，理由：
1. 数据库层面保证一致性，最可靠
2. 代码改动最小
3. 无需引入新依赖
4. 业界通用做法

### 130.3 实施方案A详细步骤

#### 步骤1：清理已有重复数据

```sql
-- 查找重复记录
SELECT user_id, account_type, COUNT(*) 
FROM accounts 
GROUP BY user_id, account_type 
HAVING COUNT(*) > 1;

-- 保留最早创建的账户（id最小），删除其他记录
DELETE FROM accounts 
WHERE id NOT IN (
    SELECT MIN(id) 
    FROM accounts 
    GROUP BY user_id, account_type
);
```

#### 步骤2：添加数据库唯一约束

```sql
ALTER TABLE accounts
ADD CONSTRAINT unique_user_balance_account
UNIQUE (user_id, account_type);
```

#### 步骤3：修改 `account_service.py`

在 `get_or_create_account` 方法中添加 `IntegrityError` 捕获：

```python
from sqlalchemy.exc import IntegrityError

async def get_or_create_account(self, user_id: str) -> Account:
    # ... 查询逻辑 ...
    
    if not account:
        account = Account(...)
        self.db.add(account)
        try:
            await self.db.commit()
            await self.db.refresh(account)
        except IntegrityError:
            # 唯一约束冲突：说明其他请求已经创建了账户
            await self.db.rollback()
            logger.info(f"[GetOrCreate] Account already exists (concurrent creation), querying...")
            # 重新查询账户
            result = await self.db.execute(
                select(Account).where(
                    and_(
                        Account.user_id == uuid.UUID(user_id),
                        Account.account_type == "balance"
                    )
                )
            )
            account = result.scalars().first()
            if not account:
                raise RuntimeError(f"账户创建失败且无法查询到账户: {user_id}")
    
    return account
```

## 问题131：recharge 充值接口使用 flush 而非 commit 导致充值后余额为0

### 131.1 问题描述

**发现时间**：2026-04-23

**现象**：
1. 用户 test9 充值后立即显示余额为 0.00 元
2. 刷新页面后余额仍然为 0
3. 充值账单记录显示成功，但余额未增加

**影响范围**：
- 充值接口：`POST /api/v1/billing/recharge`
- 所有通过该接口充值的用户

### 131.2 问题原因分析

#### 根本原因：使用了 flush() 而非 commit()

在 `billing.py` 的 `recharge` 接口中：

```python
# 修复前代码（第156行）
await db.flush()  # ❌ 问题所在！

return BaseResponse(
    data={
        "order_id": bill_no,
        "balance": float(account.balance),  # 此时在事务内存中可见
    }
)
```

**flush() vs commit() 的区别**：

| 方法 | 作用 | 数据持久化 | 其他会话可见 | 适用场景 |
|------|------|-----------|-------------|----------|
| `flush()` | 将更改写入数据库事务 | ❌ 否 | ❌ 否 | 内部测试、批量操作 |
| `commit()` | 提交事务，将更改写入数据库 | ✅ 是 | ✅ 是 | 业务操作完成 |

**问题场景**：
1. `flush()` 将数据写入当前事务的缓冲区
2. `return` 返回响应时，事务尚未提交
3. 事务结束后（请求结束时自动回滚），数据丢失
4. 用户刷新页面查询数据库，看不到充值后的余额

### 131.3 修复方案

**修改文件**：`src/api/v1/billing.py` 第91-165行

**修复前**：
```python
# 更新账户
account.balance = str(new_balance)
account.total_recharge = str(Decimal(account.total_recharge or "0") + Decimal(amount))

await db.flush()  # ❌ 只写入事务缓冲区，未持久化

return BaseResponse(
    data={
        "order_id": bill_no,
        "balance": float(account.balance),
    }
)
```

**修复后**：
```python
# 更新账户
account.balance = str(new_balance)
account.total_recharge = str(Decimal(account.total_recharge or "0") + Decimal(amount))

# ✅ 必须 commit 才能持久化数据！
await db.commit()
await db.refresh(account)  # 刷新获取最新数据
logger.info(f"[Recharge] 充值成功: user_id={current_user.id}, amount={amount}, new_balance={account.balance}")

return BaseResponse(
    data={
        "order_id": bill_no,
        "balance": float(account.balance),
    }
)
```

### 131.4 修复效果

- ✅ 充值后余额立即可见
- ✅ 刷新页面后余额保持不变
- ✅ 其他接口查询余额正确显示

## 问题132：支付回调事务一致性问题导致充值后余额为0

### 132.1 问题描述

**发现时间**：2026-04-23

**现象**：
1. 用户通过模拟支付（payment.py）充值后，余额显示为 0
2. 充值账单显示成功，但余额未增加
3. 直接充值接口（billing.py）可以正常充值

**影响范围**：
- 支付回调：`POST /api/v1/payments/callback`
- 模拟支付流程：创建订单 → 模拟支付 → 回调处理

### 132.2 问题原因分析

#### 根本原因：事务提交顺序不一致

在 `payment_service.py` 的支付回调流程中：

**原有问题流程**：
```
1. payment.status = "completed"      ← 未提交
2. add_balance() 内部执行 commit()  ← 已提交 ✅
3. user.role = "developer"           ← 未提交
4. handle_payment_callback 末尾 commit() ← 再次提交
```

**问题分析**：
1. `AccountService.add_balance()` 内部执行了 `commit()`
2. `Payment` 状态更新在 `add_balance()` 之后
3. 如果后续步骤（用户角色升级）失败：
   - `add_balance()` 已经提交 ✅ 余额已到账
   - `Payment` 状态未提交 → 回滚为 `pending`
   - 数据不一致

**另一个问题：多次 commit**
1. `get_or_create_account()` 内部有 `commit()`（账户创建时）
2. `add_balance()` 内部有 `commit()`
3. `handle_payment_callback()` 末尾有 `commit()`

这导致多个事务被分割，可能出现数据不一致。

### 132.3 修复方案

#### 设计原则：统一事务管理

**新设计**：
1. 内部服务方法（`add_balance`、`deduct_balance`、`get_or_create_account`）**只 flush，不 commit**
2. 由调用者（API 接口或业务编排层）统一管理事务提交
3. 确保所有相关操作在同一个事务中完成

**修改文件及内容**：

#### 1. `account_service.py` - `get_or_create_account` 方法

**修改前**：
```python
if not account:
    account = Account(...)
    self.db.add(account)
    try:
        await self.db.commit()  # ❌ 内部提交
        await self.db.refresh(account)
    except IntegrityError:
        await self.db.rollback()
        # 重新查询
```

**修改后**：
```python
if not account:
    account = Account(...)
    self.db.add(account)
    # ✅ 只 flush，由调用者统一提交
    await self.db.flush()
    logger.info(f"[GetOrCreate] Account staged (not committed yet): user_id={user_id}")
```

#### 2. `account_service.py` - `add_balance` 方法

**修改前**：
```python
await self.db.flush()
await self.db.commit()  # ❌ 内部提交
await self.db.refresh(account)  # ❌ 在 commit 后的 refresh
```

**修改后**：
```python
# ✅ 只 flush，由调用者统一提交
await self.db.flush()
logger.info(f"[AddBalance] Flushed: balance={account.balance}, bill_no={bill.bill_no}")
return account, bill
```

#### 3. `payment_service.py` - `handle_payment_callback` 方法

**修改后**：
```python
# 【修复】确保所有操作在同一个事务中
# 1. 更新 Payment 状态
payment.transaction_id = transaction_id
payment.pay_time = datetime.utcnow()
payment.callback_status = "processed"
# ...

if status == "success":
    payment.status = "completed"
    # 2. 在同一个事务中增加余额和升级用户
    try:
        await self._process_successful_payment(payment)
    except Exception as e:
        payment.status = "failed"
        payment.remark = f"处理失败: {str(e)}"
        # 继续执行，尝试提交失败状态

# 3. 统一提交所有更改
try:
    await self.db.commit()
    logger.info(f"[HandleCallback] Transaction committed")
except Exception as e:
    await self.db.rollback()
    raise

# 4. 刷新获取最新状态
await self.db.refresh(payment)
```

#### 4. `user.py` - `upgrade_with_payment` 接口

在调用 `deduct_balance` 后立即 commit：
```python
await account_service.deduct_balance(...)
await db.commit()  # ✅ 立即提交扣费
await db.refresh(account)
```

#### 5. `billing.py` - `recharge` 接口

添加了完整的 try-except 和环境标识：
```python
try:
    # ... 业务逻辑 ...
    await db.commit()
    await db.refresh(account)
    await db.refresh(bill)
except Exception as e:
    await db.rollback()
    raise APIError(f"充值失败: {str(e)}")
```

### 132.4 修复效果

- ✅ **事务一致性**：所有相关操作在同一事务中完成
- ✅ **原子性**：任何一步失败，所有更改回滚
- ✅ **数据可靠**：不存在"余额到账但 Payment 状态失败"的情况
- ✅ **日志完善**：详细的日志便于追踪调试

## 问题134：普通用户充值后右上角仍显示"普通用户"

### 134.1 问题描述

**发现时间**：2026-04-23

**问题现象**：
- 普通用户充值成功后，余额正确增加
- 但右上角仍然显示"普通用户"
- 左侧菜单显示开发者菜单（能正常访问）

**截图描述**：
- 右上角：显示"普通用户"
- 左侧菜单：开发者工作台、API Keys、仓库市场等（开发者菜单）
- 余额显示：¥50.00（正确更新）

### 134.2 问题原因分析

**根本原因**：前端 store 中的用户信息缓存没有及时更新

**详细分析**：

1. **用户登录时**，`useAuthStore` 使用 `persist` 中间件将用户数据持久化到 `localStorage`
2. **用户充值时**，后端更新了数据库中的 `user_type` 为 `developer`
3. **充值成功后**，虽然 Recharge 页面调用了 `authApi.me()` 获取最新用户信息并调用 `setUser()` 更新 store
4. **Layout 组件**只在挂载时获取一次用户信息（`useEffect(() => {...}, [])`），不会再次刷新
5. **结果**：右上角的"普通用户"标识来自 Layout 组件中的 `user.user_type`，而这个值是旧的缓存数据

### 134.3 解决方案

**方案：在 Layout 组件中添加定期刷新机制**

**修改文件**：`web/src/components/Layout.tsx`

**修改内容**：
```tsx
// 获取用户信息（始终从后端刷新，避免本地缓存过期）
useEffect(() => {
  const fetchUser = async () => {
    try {
      const userData = await authApi.me()
      const currentUser = useAuthStore.getState().user
      // 如果用户类型发生变化，打印日志
      if (currentUser && userData.user_type !== currentUser.user_type) {
        console.log(`[Layout] 用户类型变化: ${currentUser.user_type} -> ${userData.user_type}`)
      }
      useAuthStore.getState().setUser(userData)
    } catch (error) {
      console.error('获取用户信息失败:', error)
    } finally {
      setLoading(false)
    }
  }

  // 始终刷新用户信息，从后端获取最新数据
  fetchUser()
  
  // 【V4.0 新增】每30秒定期刷新用户信息，确保角色变化后能及时更新
  const intervalId = setInterval(fetchUser, 30000)
  return () => clearInterval(intervalId)
}, [])
```

### 134.4 验证方法

1. **重启前端服务**，使修改生效
2. **使用普通用户账号登录**
3. **充值成功后**：
   - 等待30秒，观察右上角是否更新为"开发者"
   - 或者刷新页面，观察右上角是否更新为"开发者"

## 问题133：并发场景下 _get_account_or_create 导致 UniqueViolationError

### 133.1 问题描述

**发现时间**：2026-04-23

**错误信息**：
```
IntegrityError: 重复键违反唯一约束"unique_user_balance_account"
DETAIL: 键值"(user_id, account_type)=(86ae112c-xxx, balance)" 已经存在
```

**触发场景**：新建普通用户登录时，多个 API 请求同时调用 `/user/status` 接口

### 133.2 问题原因分析

**根本原因**：`user.py` 中的 `_get_account_or_create` 函数没有处理并发竞争。

当多个请求同时到达时：
1. 请求A 查询账户，不存在，创建账户
2. 请求B 查询账户，不存在，创建账户
3. 两个请求同时尝试插入相同 (user_id, account_type) 的记录
4. 数据库唯一约束冲突，抛出 IntegrityError

**代码位置**：`src/api/v1/user.py` 第 195-236 行

### 133.3 修复方案

**核心思路**：捕获 IntegrityError，当发生冲突时回滚并重新查询已有记录。

**修改文件**：`src/api/v1/user.py`

**修复代码**：
```python
async def _get_account_or_create(db: AsyncSession, user: User) -> Account:
    """获取或创建用户账户"""
    from sqlalchemy import and_
    from sqlalchemy.exc import IntegrityError
    
    try:
        result = await db.execute(
            select(Account).where(
                and_(
                    Account.user_id == user.id,
                    Account.account_type == "balance"
                )
            )
        )
        accounts = result.scalars().all()
        
        if len(accounts) > 1:
            account = accounts[0]  # 取第一条
        elif len(accounts) == 0:
            account = None
        else:
            account = accounts[0]
        
        if not account:
            account = Account(
                user_id=user.id,
                account_type="balance",
                balance="0",
                frozen_balance="0",
                total_recharge="0",
                total_consume="0",
            )
            db.add(account)
            await db.flush()  # 先 flush，让数据库检测约束冲突
            await db.commit()
            await db.refresh(account)
        
        return account
        
    except IntegrityError:
        # 并发情况下，其他请求已创建了账户，回滚并重新查询
        logger.warning(f"[Account] Concurrent account creation detected for user {user.id}")
        await db.rollback()
        result = await db.execute(
            select(Account).where(
                and_(
                    Account.user_id == user.id,
                    Account.account_type == "balance"
                )
            )
        )
        account = result.scalars().first()
        if not account:
            raise RuntimeError(f"无法获取用户账户: {user.id}")
        return account
```

### 133.4 关键点

1. **先 flush 再 commit**：让数据库先检测约束冲突
2. **捕获 IntegrityError**：表示其他请求已创建了账户
3. **rollback 并重新查询**：获取已存在的账户
4. **不要在 except 中再次 commit**：避免嵌套事务问题

```python
    # 没有账户，不扣费，但抛出余额不足异常阻止调用
    raise HTTPException(
        status_code=402,
        detail=f"账户不存在或余额为0，无法调用API。请先充值。"
    )

# 检查余额是否足够
balance = Decimal(str(account.balance))
if balance < cost:
    # 余额不足，拒绝API调用
    raise HTTPException(
        status_code=402,
        detail=f"余额不足。当前余额：¥{balance:.2f}，本次调用需要：¥{cost:.2f}。请先充值。"
    )
```
