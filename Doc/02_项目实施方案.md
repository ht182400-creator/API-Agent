## 第一章 实施方案概述

### 1.1 方案目标

本方案旨在制定详细的技术实施路线，确保API服务改造项目按时、按质、按量交付。方案涵盖技术选型、架构设计、实施步骤、资源配置、风险管理等关键内容。

### 1.2 技术选型依据

**选型原则**：
1. 成熟稳定：优先选择经过大规模验证的技术组件
2. 社区活跃：有活跃的社区支持和丰富的学习资源
3. 易于运维：降低运维复杂度，减少维护成本
4. 成本可控：优先选择开源或低成本方案

**技术栈对比分析**：

| 组件类型 | 备选方案 | 推荐方案 | 推荐理由 |
|----------|----------|----------|----------|
| 后端框架 | Flask/FastAPI/Django | **FastAPI** | 高性能、自动文档、类型安全 |
| 数据库 | MySQL/PostgreSQL | PostgreSQL | 功能强大、数据完整性强 |
| 缓存 | Redis/Memcached | Redis | 支持丰富的数据结构 |
| 对象存储 | MinIO/AWS S3 | MinIO | S3兼容、部署灵活 |
| 反向代理 | Nginx/Apache | Nginx | 高性能、稳定、配置灵活 |
| 容器化 | Docker/Kubernetes | Docker | 成熟、易用、普及率高 |

#### 1.3.2 详细优劣势对照表

| Starlette生态继承 | 相对Django缺ORM/Admin |

| 微服务架构成熟 | 类型安全不足 |

| 优势 | 劣势 |

#### 1.3.6 本项目选型结论

**最终推荐：Python + FastAPI**

**选型理由**：
1. 课程改造项目追求开发效率和易维护性
2. 自动API文档对前后端协作极有价值
3. Python AI生态为未来功能扩展预留空间
4. 类型安全保证代码质量
5. 团队学习曲线平缓，快速上手

**不选其他方案原因**：
- Node.js：动态类型风险高，不适合严肃API服务
- Java Spring Boot：过于笨重，开发效率低
- Go：性能优异但学习成本高，适合进阶

### 2.3 核心模块设计

#### 2.3.1 认证模块

**流程设计**：
```
┌─────────────────────────────────────────────────────────────────────┐
│                         JWT认证流程                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  用户登录                                                            │
│  1. 接收email + password                                            │
│  2. 验证密码（bcrypt）                                              │
│  3. 生成JWT Token                                                   │
│     - access_token (1小时有效期)                                    │
│     - refresh_token (7天有效期)                                      │
│  4. 返回Token给客户端                                                │
│                                                                      │
│  API请求认证                                                         │
│  1. 从Header提取Authorization                                       │
│  2. 验证Token签名                                                   │
│  3. 检查Token有效期                                                 │
│  4. 验证用户状态                                                    │
│  5. 通过则放行，失败返回401                                          │
│                                                                      │
│  Token刷新                                                          │
│  1. 接收refresh_token                                               │
│  2. 验证refresh_token                                               │
│  3. 生成新的access_token                                            │
│  4. 可选：延长refresh_token有效期                                    │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

**关键代码结构**：
```python
# auth_service.py
class AuthService:
    def login(self, email: str, password: str) -> TokenPair:
        """用户登录，返回Token对"""
        
    def verify_token(self, token: str) -> TokenPayload:
        """验证Token，返回payload"""
        
    def refresh_token(self, refresh_token: str) -> TokenPair:
        """刷新Token"""
        
    def revoke_token(self, token: str) -> bool:
        """撤销Token"""
```

#### 2.3.2 API Key模块

**Key生成算法**：
```
前缀(4) + 分隔符(1) + 主体(27) + 校验(4) = 32位
格式：APIK_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

**验证流程**：
```
1. 接收API Key (Header: X-API-Key)
2. 验证Key格式
3. 查询数据库验证Key有效性
4. 验证Key状态（未过期、未禁用）
5. 验证租户状态
6. 检查调用配额
7. 记录调用日志
8. 返回业务响应
```

#### 2.3.3 限流模块

**限流策略**：
```yaml
限流配置:
  default:
    qps: 100          # 每秒请求数
    daily: 10000      # 每日请求数
    burst: 20         # 突发容量
    
  租户等级:
    basic:
      qps: 50
      daily: 5000
    standard:
      qps: 100
      daily: 10000
    premium:
      qps: 200
      daily: 50000
```

**限流实现**：
- 使用Redis + 滑动窗口算法
- 支持租户级别和Key级别限流
- 限流后返回429状态码和Retry-After头

from alipay import AliPay

alipay = AliPay(

    app_notify_url=settings.ALIPAY_NOTIFY_URL,
    app_private_key_string=settings.ALIPAY_PRIVATE_KEY,
    alipay_public_key_string=settings.ALIPAY_PUBLIC_KEY,
)

if alipay.verify(request_params, signature):
    # 处理支付成功逻辑
    pass
```

**微信支付回调验证**：
```python
from wechatpayv3 import WeChatPay

wxpay = WeChatPay(

    apiv3_key=settings.WECHAT_API_KEY,
    mchid=settings.WECHAT_MCHID,
    private_key=settings.WECHAT_PRIVATE_KEY,
    cert_serial_no=settings.WECHAT_CERT_SERIAL_NO,
    certs=certs,
)

verified = wxpay.verify(
    signature, timestamp, nonce, request_body
)
if verified:
    # 处理支付成功逻辑
    pass
```

#### 9.4.2 对账流程

```
┌─────────────────────────────────────────────────────────────────┐
│                       对账操作流程                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  [1. 选择日期] ──> [2. 选择渠道] ──> [3. 执行对账]             │
│         │              │              │                        │
│         v              v              v                        │
│   选择需要对账    支付宝/微信/     调用对账API                    │
│   的日期(默认    银行卡/全部      执行数据比对                    │
│   昨日)                                               │          │
│                                             ┌────────┴────────┐ │
│                                             │   对账结果       │ │
│                                             ├──────────────────┤ │
│                                             │ ✅ 完全匹配     │ │
│                                             │ ⚠️ 本地多(长款)│ │
│                                             │ ❌ 第三方多(短款)│ │
│                                             │ ⚡ 金额不一致   │ │
│                                             └────────┬─────────┘ │
│                                                      │            │
│                                                      ▼            │
│                                             差异项自动生成      │
│                                             到 reconciliation_disputes表
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

#### 9.4.3 差异类型处理

| 差异类型 | 本地记录 | 第三方记录 | 处理方式 |
|----------|----------|------------|----------|
| match | ✅ 有 | ✅ 有且金额相同 | 无需处理 |
| long | ✅ 有 | ❌ 无 | 核实是否重复支付或掉单 |
| short | ❌ 无 | ✅ 有 | 可能是掉单，需补录 |
| amount_diff | ✅ 有 | ✅ 有但金额不同 | 核查金额差异原因 |
