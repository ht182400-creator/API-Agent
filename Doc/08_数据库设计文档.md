## 第一章 数据库概述

### 1.1 设计目标

| 目标 | 说明 | 量化指标 |
|------|------|----------|
| 性能 | 高并发读写支持 | 单表10万+ QPS |
| 可用性 | 主从架构 | 99.9% 可用性 |
| 扩展性 | 水平扩展能力 | 支持分库分表 |
| 安全性 | 多层安全防护 | 零数据泄露 |
| 可维护性 | 清晰的表结构 | 开发效率提升 |

### 1.2 设计原则

| 原则 | 说明 | 实践 |
|------|------|------|
| 规范化 | 遵循3NF范式 | 减少数据冗余，避免更新异常 |
| 完整性 | 约束确保准确性 | 主键、外键、CHECK约束 |
| 安全性 | 敏感数据保护 | 加密存储、权限隔离 |
| 可扩展性 | 预留扩展字段 | JSONB、灵活字段 |
| 性能优先 | 索引与分区 | 按需分区、读写分离 |
| 审计追溯 | 完整操作记录 | 审计日志、操作追踪 |

### 1.3 技术选型

| 组件 | 版本 | 说明 | 选型理由 |
|------|------|------|----------|
| PostgreSQL | 15+ | 主数据库 | 功能强大、JSON支持好、扩展性强 |
| Redis | 7+ | 缓存层 | 高性能、支持丰富数据结构 |
| MinIO | Latest | 对象存储 | S3兼容、部署简单 |

## 第四章 性能分析与优化

### 4.1 查询性能预估

| 查询场景 | 表名 | 数据量级 | 预估QPS | 优化方案 |
|----------|------|----------|---------|----------|
| API Key验证 | api_keys | 10万 | 5000+ | 索引+Redis缓存 |
| Token验证 | users | 1万 | 2000+ | Redis缓存 |
| 日志写入 | api_logs | 亿级 | 10000 | 分区+批量写入 |
| 日志查询 | api_logs | 月分区 | 100 | 索引+分区裁剪 |
| 统计聚合 | api_request_stats | 百万级 | 500 | 预聚合 |
| 租户查询 | tenants | 1000 | 100 | 索引 |

### 4.2 性能优化矩阵

| 优化项 | 实现方式 | 预期收益 | 实施难度 |
|--------|----------|----------|----------|

```python
async def validate_api_key(api_key: str) -> Optional[APIKey]:
    cache_key = f"apikey:{api_key}"
    
    # 1. 先查缓存
    cached = await redis.get(cache_key)
    if cached:
        return APIKey.model_validate_json(cached)
    
    # 2. 缓存未命中，查数据库
    key = await db.query(
        APIKey
    ).filter(
        APIKey.api_key == api_key,
        APIKey.status == "active",
        or_(
            APIKey.expires_at.is_(None),
            APIKey.expires_at > datetime.utcnow()
        )
    ).first()
    
    if key:
        # 3. 写入缓存
        await redis.setex(
            cache_key,
            600,  # 10分钟
            key.model_dump_json()
        )
    
    return key
```

#### 实现3: 限流计数器

```python
# Redis滑动窗口限流
async def check_rate_limit(api_key_id: str, limit: int, window: int) -> bool:
    key = f"ratelimit:{api_key_id}"
    now = time.time()
    window_start = now - window
    
    pipe = redis.pipeline()
    pipe.zremrangebyscore(key, 0, window_start)
    pipe.zadd(key, {str(now): now})
    pipe.zcard(key)
    pipe.expire(key, window)
    results = await pipe.execute()
    
    return results[2] <= limit
```

### 13.1 问题1：充值回调未调用导致余额和账单未更新

**问题描述**：
用户充值后，模拟支付成功，但账户余额没有变化，账单明细也没有充值记录。

**根本原因**：
充值流程需要两步：
1. 创建支付订单：`POST /api/v1/payments/create`
2. 调用回调接口：`POST /api/v1/payments/callback` → 增加余额 + 创建账单

前端模拟支付成功后，只更新了本地状态，没有调用后端回调接口。

**解决方案**：
1. 前端添加 `mockPaymentCallback` 方法调用回调接口
2. 后端删除 `_process_successful_payment` 中重复的 Bill 创建

## 第十一章 开发问题记录 (V2.3新增)

### 11.1 概述

本章记录仓库管理模块数据库设计开发过程中遇到的问题和解决方案。

### 11.2 问题1：SLA字段存储格式

**问题描述**：
后端 `sla_uptime` 字段存储为字符串格式 `"99.9%"`（包含百分号），前端获取后需要进行类型转换。

**数据库字段定义**：

```sql
sla_uptime VARCHAR(10)  -- 存储格式: "99.9%"
sla_latency_p99 INTEGER -- 存储格式: 数字（毫秒）
```

**解决方案**：

1. **数据库层**：保持现有字符串格式存储
2. **API 层**：在序列化时进行格式转换
```python
# repositories.py 中 SLA 转换
uptime=float(repo.sla_uptime.replace('%', '')) if repo.sla_uptime else None
```

### 11.3 问题2：Repository 表字段命名

**问题描述**：
数据库表使用 `repo_type` 字段，但前端期望 `type` 字段。

**数据库字段定义**：

```sql
repo_type VARCHAR(50)  -- 仓库类型: psychology, translation, vision, stock, ai, custom
```

**解决方案**：

在 API 响应中使用别名映射：
```python
# repositories.py 中字段映射
"type": repo.repo_type,  # API 响应使用 type 别名
```

| repo_id | UUID | 仓库ID（NULL表示全局默认配置） |
| repo_name | VARCHAR(100) | 仓库名称 |
| pricing_type | VARCHAR(20) | 计费类型：call（按次）/ token（按Token）/ package（套餐） |
| price_per_call | DECIMAL(10,4) | 单次调用价格 |
| price_per_1k_input_tokens | DECIMAL(10,4) | 每千输入Token价格 |
| price_per_1k_output_tokens | DECIMAL(10,4) | 每千输出Token价格 |
| package_price | DECIMAL(10,2) | 套餐价格 |
| package_calls | INTEGER | 套餐包含调用次数 |
| package_tokens | INTEGER | 套餐包含Token数量 |
| is_active | BOOLEAN | 是否启用 |
| priority | INTEGER | 优先级 |

### 17.4 计费类型说明

| 类型 | 说明 | 计算公式 |
|------|------|----------|
| call | 按调用次数计费 | 消费金额 = 调用次数 × price_per_call |
| token | 按Token数量计费 | 消费金额 = (输入Token ÷ 1000) × price_per_1k_input + (输出Token ÷ 1000) × price_per_1k_output |
| package | 按套餐包计费 | 预付费购买套餐，按套餐权益扣减 |

### 17.5 设计说明

1. **全局配置**：
   - `repo_id = NULL` 的记录为全局默认配置
   - 没有特定仓库配置时使用全局配置

2. **优先级机制**：
   - `priority` 字段支持按仓库配置优先级
   - 数字越大优先级越高

3. **灵活计费**：
   - 支持三种计费模式切换
   - 未使用的字段可以为空或0
