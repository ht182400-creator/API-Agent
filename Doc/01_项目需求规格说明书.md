## 第一章 项目概述

### 1.1 项目背景

本项目旨在将现有的课程项目代码仓库改造成可对外交付的商业化API服务产品。随着产品化需求的明确，需要建立完整的API服务体系，包括用户认证、权限管理、业务接口、管理后台、监控运维等能力，以支撑未来的商业化运营。

### 1.2 项目目标

**总体目标**：在12周内完成从代码仓库到可交付API服务产品的改造，建立完整的运营支撑体系。

**具体目标**：

| 目标编号 | 目标描述 | 量化指标 |
|----------|----------|----------|
| OBJ-001 | 完成代码仓库Docker容器化 | 镜像构建成功率100% |
| OBJ-002 | 实现标准化的API服务体系 | 支持JWT+API Key双认证 |
| OBJ-003 | 建立完整的部署架构 | 支持Docker Compose一键部署 |
| OBJ-004 | 开发管理后台 | 支持租户、API Key、统计管理 |
| OBJ-005 | 交付完整文档体系 | 部署文档、API文档、运维手册 |
| OBJ-006 | 通过验收测试 | 测试覆盖率>=70%，P0用例100%通过 |

### 1.3 项目范围

**纳入范围**：
- 现有代码仓库的Docker容器化改造
- 用户认证模块（JWT）
- API Key管理与鉴权
- 核心业务API接口
- Nginx反向代理网关
- PostgreSQL数据库集成
- Redis缓存集成
- MinIO对象存储集成
- 管理后台开发
- 部署架构设计与实施
- 文档体系建设

**不纳入范围**：
- 原有课程项目业务逻辑的大规模重构
- 移动端应用开发
- 多语言国际化
- 高端商业功能（如计费结算）

## 第二章 业务需求详述

### 2.1 用户角色分析

| 角色 | 描述 | 主要操作 |
|------|------|----------|
| API调用方 | 使用API服务的外部开发者或系统 | 注册、申请Key、调用API、查看使用统计 |
| 租户管理员 | 管理租户内的用户和资源 | 租户用户管理、Key分配、权限配置 |
| 系统管理员 | 负责整个系统的运维管理 | 系统配置、监控查看、问题处理 |
| 开发者 | 进行API调用开发的人员 | 阅读文档、测试API、集成开发 |

### 2.2 业务场景

**场景一：外部开发者接入**
张三是一名第三方开发者，希望接入本API服务为自己的应用提供能力支持。他首先需要注册账号，申请API Key，然后阅读API文档，在他的应用中集成API调用。系统会记录他的使用量，他可以通过后台查看调用统计。

**场景二：多租户管理**
某企业A购买了API服务，需要给内部多个开发团队分配不同的访问权限。租户管理员可以创建子账号，为每个子账号分配不同的API Key，设置不同的调用配额，实现灵活的权限管理。

**场景三：运营监控**
系统管理员需要实时监控系统运行状态，包括API响应时间、错误率、各租户使用量等。当出现异常时，需要能够快速定位问题，查看相关日志。

### 2.3 业务流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                        API服务使用流程                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  1. 注册/登录                                                       │
│     填写信息 ──► 验证邮箱 ──► 创建账号 ──► 登录系统                  │
│                                                                      │
│  2. API Key申请                                                    │
│     登录 ──► 进入控制台 ──► 申请API Key ──► 获取Key和Secret        │
│                                                                      │
│  3. API调用                                                         │
│     阅读文档 ──► 获取调用示例 ──► 集成到应用 ──► 调用API            │
│                                                                      │
│  4. 使用统计                                                        │
│     进入控制台 ──► 查看调用统计 ──► 下载报表                        │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

## 第三章 功能需求详述

### 3.1 用户认证模块

**FR-AUTH-001 用户注册**
- 支持邮箱注册
- 密码强度校验（至少8位，包含大小写字母和数字）
- 注册后发送验证邮件
- 邮箱验证后才可使用

**FR-AUTH-002 用户登录**
- 邮箱+密码登录
- 登录失败锁定（连续5次失败锁定30分钟）
- JWT Token生成（access_token有效期1小时，refresh_token有效期7天）

**FR-AUTH-003 Token管理**
- Token刷新机制
- Token撤销功能
- 多设备登录管理

### 3.2 API Key管理模块

**FR-KEY-001 Key生成**
- 自动生成32位唯一Key
- 生成对应Secret用于签名验证
- 支持自定义Key名称和备注

**FR-KEY-002 Key管理**
- 查看已申请的Key列表
- 设置Key的有效期
- 启用/禁用Key
- 删除Key
- 每个租户最多支持10个Key

**FR-KEY-003 Key认证**
- API Key + Secret签名验证
- 支持在请求头或参数中传递Key
- 支持签名时效性校验（防止重放攻击）

### 3.3 核心业务API模块

**FR-API-001 RESTful API设计**
- 统一的URL规范：/api/v1/{resource}
- 统一的请求格式：JSON
- 统一的响应格式：{code, message, data}
- 标准的HTTP状态码使用

**FR-API-002 API文档**
- Swagger/OpenAPI 3.0规范
- 自动从代码生成文档
- 支持在线调试
- 支持文档导出

**FR-API-003 限流机制**
- 基于API Key的限流
- 默认QPS限制：100次/秒
- 默认日调用限制：10000次
- 限流后返回429状态码

### 3.4 管理后台模块

**FR-ADMIN-001 租户管理**
- 租户列表查看
- 租户创建、编辑、禁用
- 租户资源配置（QPS、调用量限制）
- 租户使用量统计

**FR-ADMIN-002 用户管理**
- 租户内用户列表
- 用户创建、编辑、禁用
- 角色分配
- 权限配置

**FR-ADMIN-002a 超级管理员用户管理（V2.4新增）**
- 查看所有用户列表（支持分页、搜索、筛选）
- 更新用户信息（用户类型、角色、状态、VIP等级、权限）
- 禁用/启用用户
- 删除用户
- 查看用户详细信息和操作日志

**FR-ADMIN-002b 管理员用户管理（V2.4新增）**
- 查看普通用户列表（不包括管理员和超级管理员）
- 更新用户状态（禁用/启用）
- 更新用户VIP等级
- 删除普通用户（不包括管理员和超级管理员）
- 权限限制：不能修改管理员或超级管理员的账户

**FR-ADMIN-003 统计报表**
- 实时调用量统计
- 按时间维度汇总（日/周/月）
- 按接口维度统计
- 按租户维度统计
- 导出Excel报表

### 3.5 系统管理模块

**FR-SYS-001 系统配置**
- API版本管理
- 全局限流配置
- 邮件服务配置
- 存储配置

**FR-SYS-002 日志管理**
- API调用日志
- 错误日志
- 操作审计日志
- 日志查询和导出

### 3.6 通知模块（V1.1新增）

**FR-NOTIFY-001 通知中心**
- 用户站内通知列表展示
- 支持分页查询
- 未读/已读分类查看
- 单条和全部标记已读
- 单条和批量删除通知

**FR-NOTIFY-002 通知下拉面板**
- 顶部导航栏铃铛图标
- 实时显示未读数量Badge
- 下拉面板展示最近5条通知
- 点击通知标记已读并跳转
- 支持"全部已读"快捷操作

**FR-NOTIFY-003 通知类型**
- 系统通知（system）- 普通优先级
- 账单通知（billing）- 高优先级
- API通知（api）- 普通优先级
- 安全通知（security）- 高/紧急优先级

**FR-NOTIFY-004 通知偏好设置**
- 用户可配置是否接收邮件通知
- 用户可配置是否接收站内通知
- 用户可配置各类型通知的接收开关
- 偏好设置持久化存储

**FR-NOTIFY-005 管理员通知功能**
- 管理员可为指定用户创建通知
- 管理员可广播通知给所有用户
- 支持按用户类型筛选广播范围

### 3.7 仓库管理模块（V2.0新增）

**FR-REPO-001 仓库浏览**
- 查看可用API仓库列表
- 支持按类型、协议筛选
- 显示仓库基本信息（名称、描述、定价、限流）
- 显示仓库提供的API端点列表

**FR-REPO-002 仓库端点配置**
- 每个仓库可配置多个API端点
- 端点存储到 `repo_endpoints` 表
- 支持配置：路径、方法、描述、分类
- 支持端点级限流设置
- 支持显示顺序自定义

**FR-REPO-003 仓库限流配置**
- 每个仓库可配置独立的限流策略
- 限流配置存储到 `repo_limits` 表
- 支持多级限流：分钟/小时/日
- 支持突发限制和并发限制
- 支持配额上限设置

**FR-REPO-004 仓库详情查询**
- 获取仓库完整信息
- 从数据库读取端点列表
- 从数据库读取限流配置
- 显示定价策略和SLA信息

**FR-REPO-005 仓库所有者管理**
- 仓库所有者可创建自己的仓库
- 支持仓库的增删改查
- 仓库状态管理（待审核/已上线/已下线）

## 第六章 接口规格

### 6.1 API基础规范

**基础URL**：https://api.example.com

**认证方式**：
1. Header方式：`Authorization: Bearer {access_token}`
2. API Key方式：`X-API-Key: {api_key}`

**通用请求头**：
```
Content-Type: application/json
Accept: application/json
X-Request-ID: {uuid}
X-API-Key: {api_key}
X-Signature: {hmac_signature}
X-Timestamp: {unix_timestamp}
```

**通用响应格式**：
```json
{
  "code": 200,
  "message": "success",
  "data": {},
  "request_id": "uuid",
  "timestamp": 1713200000
}
```

**错误码规范**：
| 错误码 | 说明 |
|--------|------|
| 200 | 成功 |
| 400 | 请求参数错误 |
| 401 | 未认证或认证失败 |
| 403 | 无权限访问 |
| 404 | 资源不存在 |
| 429 | 请求过于频繁 |
| 500 | 服务器内部错误 |

### 6.2 主要API接口

**认证接口**：
| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/v1/auth/register | 用户注册 |
| POST | /api/v1/auth/login | 用户登录 |
| POST | /api/v1/auth/refresh | 刷新Token |
| POST | /api/v1/auth/logout | 登出 |

**API Key接口**：
| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/v1/keys | 创建API Key |
| GET | /api/v1/keys | 获取Key列表 |
| GET | /api/v1/keys/{id} | 获取Key详情 |
| PUT | /api/v1/keys/{id} | 更新Key |
| DELETE | /api/v1/keys/{id} | 删除Key |

## 第十章 仓库管理模块开发说明 (V2.1新增)

### 10.1 模块概述

仓库管理模块为平台提供 API 仓库的浏览、创建、更新和删除功能。包括：

- 仓库市场：供开发者浏览和发现可用的API服务
- 仓库详情：展示仓库的端点、限流、定价等信息
- 仓库管理：供仓库所有者管理自己的仓库

### 10.2 前端页面

| 页面 | 路径 | 功能 |
|------|------|------|
| 仓库市场 | `/developer/repos` | 浏览所有可用仓库 |
| 仓库详情 | `/developer/repos/:slug` | 查看仓库详细信息 |
| 仓库管理 | `/owner/repos` | 管理我的仓库 |

### 10.3 数据字段规范

#### 仓库类型 (repo_type)

| 值 | 说明 |
|----|------|
| psychology | 心理问答 |
| translation | 翻译服务 |
| vision | 图像识别 |
| stock | 股票行情 |
| ai | AI服务 |
| custom | 自定义 |

#### 协议类型 (protocol)

| 值 | 说明 |
|----|------|
| http | HTTP REST API |
| grpc | gRPC |
| websocket | WebSocket |

#### 仓库状态 (status)

| 值 | 说明 |
|----|------|
| pending | 待审核 |
| approved | 已审核（待上线） |
| rejected | 已拒绝 |
| online | 已上线 |
| offline | 已下线 |

### 10.4 SLA字段说明

仓库的 SLA 信息包含以下字段：

| 字段 | 类型 | 说明 | 格式示例 |
|------|------|------|----------|
| uptime | string | 可用性目标 | "99.9%" |
| latency_p99 | number | P99延迟目标(毫秒) | 500 |

### 10.5 开发注意事项

1. **前后端字段映射**：
   - API Schema 使用 `repo_type`
   - API 响应使用 `type` 作为别名
   - 前端显示统一使用 `type`

2. **类型转换**：
   - SLA uptime 字段存储为字符串（含百分号）
   - 显示时需要进行类型检查和格式化

3. **初始状态**：
   - 新创建的仓库状态为 `pending`（待审核）
   - 需要管理员审核后才能上线

### 10.7 管理员审核流程（V2.2新增）

#### 10.7.1 审核流程概述

仓库管理采用管理员审核机制，确保平台内容质量：

```
┌─────────────────┐
│   仓库所有者    │
│   创建仓库      │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ pending         │ ← 初始状态
│ 待审核状态      │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│   管理员审核    │
│ /admin/repos    │
└────────┬────────┘
         │
    ┌────┴────┐
    │         │
    ▼         ▼
┌────────┐ ┌────────┐
│approved│ │rejected│
│已审核  │ │已拒绝  │
└───┬────┘ └────────┘
    │         │
    ▼         │
┌────────┐    │
│ online │    │
│ 已上线 │    │
└───┬────┘    │
    │         │
    ▼         │
┌────────┐    │
│offline │◄───┘
│ 已下线 │  (可重新审核)
└────────┘
```

#### 10.7.2 审核权限

| 用户类型 | 权限 | 说明 |
|----------|------|------|
| super_admin | repo:approve, repo:manage | 超级管理员，拥有全部权限 |
| admin | repo:approve, repo:manage | 管理员，拥有审核权限 |
| developer | repo:write, repo:delete | API开发者，只能创建/编辑/删除自己仓库，不能修改状态 |
| user | repo:read | 普通用户，只能浏览已上线仓库 |

> **V4.0 更新说明**：`owner` 不再是独立角色，是 `developer + 有仓库` 的业务状态。

#### 10.7.3 管理员 API

| 接口 | 方法 | 说明 | 权限 |
|------|------|------|------|
| `/repositories/admin/all` | GET | 获取所有仓库（支持状态筛选） | admin+ |
| `/repositories/{id}/approve` | POST | 审核通过 | admin+ |
| `/repositories/{id}/reject` | POST | 审核拒绝 | admin+ |
| `/repositories/{id}/online` | POST | 上线仓库 | admin+ |
| `/repositories/{id}/offline` | POST | 下线仓库 | admin+ |

#### 10.7.4 前端页面

- **管理员页面**：`/admin/repos`
  - 按状态筛选（全部/待审核/已审核/已上线/已下线/已拒绝）
  - 统计各状态仓库数量
  - 审核通过/拒绝操作（可填写备注/原因）
  - 上线/下线操作

- **开发者页面**：`/developer/repos`
  - 查看自己创建的仓库及状态
  - 创建/编辑/删除仓库
  - **不能**自行上下线（必须通过管理员审核）

## 第十二章 限流检查模块 (V2.5新增)

### 12.1 模块概述

限流检查模块在API Key验证时自动进行多级限流检查：

- **每分钟请求数 (RPM)**：防止突发流量
- **每小时请求数 (RPH)**：限制长期使用
- **日配额 (Daily Quota)**：每日调用上限
- **月配额 (Monthly Quota)**：每月调用上限
- **账户余额检查**：余额不足时警告或拦截

### 12.2 功能需求

#### FR-RATE-001 RPM限流

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| rpm_limit | 每分钟请求数上限 | 1000 |

#### FR-RATE-002 RPH限流

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| rph_limit | 每小时请求数上限 | 10000 |

#### FR-RATE-003 配额检查

| 功能 | 描述 | 说明 |
|------|------|------|
| 日配额检查 | 检查今日已使用量 | 超出限制返回429 |
| 月配额检查 | 检查本月已使用量 | 超出限制返回429 |
| 配额记录 | 每次调用记录配额使用 | 写入quota表 |

#### FR-RATE-004 余额检查

| 功能 | 描述 | 说明 |
|------|------|------|
| 余额阈值 | 低于阈值时警告 | 默认1元 |

   - `_check_rate_limit()` 方法实现多级限流检查

### 17.4 技术实现

- 格式：CSV（UTF-8 with BOM）
- 传输：StreamingResponse 流式下载
- 字段：日期/订单号/渠道/差异类型/金额/状态

## 第十八章 计费系统优化 (V3.0新增)

### 18.1 问题背景

用户反馈账单中心的"本月调用次数"和"本月消费金额"对应关系不清晰，计费规则无法配置。

**当前问题**：
- `consumption_count` 是账单记录数，不是API调用次数
- 缺少Token使用量统计
- 无法按仓库配置不同计费费率

### 18.2 优化目标

1. **精确统计**：区分API调用次数和Token使用量
2. **灵活计费**：支持按调用次数、按Token、套餐包三种计费模式
3. **可配置**：管理员可配置各仓库的计费规则

| 平均费用计算 | 计算单次调用/单Token平均费用 | 🔄 待优化 |

#### FR-BILLING-007 计费配置管理

| 功能 | 描述 | 状态 |

### 18.4 计费模式

```
         │
         ▼

         │
         ▼
   ┌─────────────┐
   │  审核队列   │  管理员可见
   └──────┬──────┘
          │
    ┌─────┴─────┐
    │           │
    ▼           ▼
  通过        拒绝
    │           │
    ▼           ▼
status=       通知用户

    │
    ▼
status=online
    │
    ▼

```
