## 问题126：Owner平台数据分析页面显示"暂无数据"

### 126.1 问题描述

**发现时间**：2026-04-22

Owner平台的数据分析页面（`/owner/analytics`）显示"暂无数据"，但实际上工作台页面有数据显示。

**表现**：
- 数据分析页面所有图表都显示"暂无数据"
- 统计卡片显示为0
- 浏览器控制台显示 API 返回数据为 `undefined`

**控制台错误日志**：
```
[Analytics] 获取到 overviewData: undefined
[Analytics] 获取到 weekly: undefined
[Analytics] 获取到 hourly: undefined
[Analytics] 获取到 sources: undefined
```

### 126.2 问题原因分析

经过深入排查，发现问题出在**前端 API 调用代码**上。

#### 根本原因：重复访问 `.data` 属性

**文件**：`api-platform/web/src/api/analytics.ts`

**问题代码**：
```typescript
export const analyticsApi = {
  getOverview: async (): Promise<OverviewStats> => {
    const response = await api.get('/owner/overview')
    return response.data  // 错误：api.get() 已经返回了 data，这里又访问 .data
  },
  // ... 其他方法同样问题
}
```

**原因说明**：

1. `client.ts` 中的 axios 拦截器已经自动提取了响应中的 `data` 字段：
```typescript
// client.ts 第 96-97 行
if (data?.data !== undefined) {
  extractedData = data.data  // 自动提取 data 字段
}

// api.get() 方法返回的就是 extractedData
return client.get(url, { params, ...config }).then((res) => (res as any).data)
```

2. `analytics.ts` 中又访问了 `.data`，导致返回 `undefined`

3. 正确做法是直接返回 `api.get()` 的结果

### 126.3 解决方案

**修改文件**：`api-platform/web/src/api/analytics.ts`

**修复后代码**：
```typescript
export const analyticsApi = {
  /**
   * 获取总览统计
   */
  getOverview: async (): Promise<OverviewStats> => {
    // api.get() 已由 client.ts 拦截器自动提取 data 字段
    return await api.get<OverviewStats>('/owner/overview')
  },

  /**
   * 获取每周统计
   */
  getWeeklyStats: async (weeks: number = 1): Promise<WeeklyStats[]> => {
    return await api.get<WeeklyStats[]>('/owner/weekly', { params: { weeks } })
  },

  /**
   * 获取24小时分布
   */
  getHourlyStats: async (): Promise<HourlyStats[]> => {
    return await api.get<HourlyStats[]>('/owner/hourly')
  },

  /**
   * 获取调用来源分布
   */
  getSourceStats: async (): Promise<SourceStats[]> => {
    return await api.get<SourceStats[]>('/owner/sources')
  }
}
```

### 126.4 排查过程记录

#### 步骤1：验证后端 API 正常

通过直接调用后端 API 验证数据是否正常：

#### 步骤2：检查前端网络请求

打开浏览器开发者工具 -> Network 标签，刷新页面。

**发现**：没有 `/api/v1/owner/overview` 的请求记录。

**原因分析**：由于 API 调用返回 `undefined`，但没有抛出错误，所以页面静默失败。

#### 步骤3：添加前端调试日志

在 `Analytics.tsx` 中添加调试日志：

#### 步骤4：检查 axios 拦截器

#### 步骤5：定位问题代码

### 126.5 相关代码位置

| 文件路径 | 修改内容 |
|---------|---------|
| `api-platform/web/src/api/analytics.ts` | 修复 API 方法返回值，移除多余的 `.data` 访问 |

### 126.6 经验总结

1. **axios 拦截器自动提取数据**：`client.ts` 中的响应拦截器会自动从 `{ code: 0, data: {...} }` 中提取 `data` 字段，所以 `api.get()` 返回的就是业务数据。

2. **API 模块不需要二次提取**：所有调用 `api.get()` 或 `api.post()` 的 API 模块，都应该直接返回结果，不需要再访问 `.data`。

3. **添加调试日志的重要性**：当问题难以定位时，在关键位置添加 `console.log` 可以快速缩小问题范围。

4. **检查 Network 请求**：如果 API 没有返回数据，首先检查 Network 标签是否有对应的请求记录。

## 问题130：PostgreSQL GROUP BY 语法错误

### 130.1 问题描述

analytics API 返回 500 错误，日志显示 PostgreSQL 错误：

```
column "api_call_logs.source" must appear in the GROUP BY clause
```

### 130.2 问题原因

SQLAlchemy 生成的 SQL 语句中，`ORDER BY` 使用了聚合函数但未在 `GROUP BY` 中正确引用。

### 130.3 解决方案

使用变量引用确保 GROUP BY 与 SELECT 表达式一致：

```python
# 定义 source 列的表达式（复用确保一致性）
source_expr = func.coalesce(APICallLog.source, 'unknown').label("source")

query = select(
    source_expr,
    func.count(APICallLog.id).label("count")
).join(
    Repository, APICallLog.repo_id == Repository.id
).where(
    Repository.owner_id == current_user.id
).group_by(
    source_expr  # 直接使用同一表达式引用
)

# 在应用层按 count 降序排序（最安全的方式）
rows = sorted(rows, key=lambda x: x.count, reverse=True)
```

### 130.4 关键要点

1. **使用变量引用**：将 `source_expr` 定义为变量，在 SELECT 和 GROUP BY 中使用同一个引用
2. **应用层排序**：避免在 SQL 中使用 `ORDER BY func.count()`，改用 Python 的 `sorted()` 在应用层排序
3. **统一时区**：所有时间处理统一使用 `datetime.now()` 而不是 `datetime.utcnow()`

## 问题133：总收益数据来源错误

### 133.1 问题描述

**发现时间**：2026-04-22

Owner 平台数据分析页面显示的"总收益"数据不正确，显示为固定的测试数据（100元），而不是从实际 API 调用记录计算。

### 133.2 问题原因

**根本原因**：`analytics.py` 中总收益的计算逻辑错误。

原始代码从 bills 表计算收益，但：
1. `bills.amount` 字段是字符串类型，无法直接 SUM
2. bills 表存储的是充值记录，不是 API 调用收益
3. 应从 `api_call_logs.cost` 字段计算

### 133.3 解决方案

**修改文件**：`api-platform/src/api/v1/analytics.py`

```python
# 获取总收益 (从 api_call_logs 的 cost 字段计算)
revenue_query = select(
    func.coalesce(func.sum(func.cast(APICallLog.cost, Numeric)), 0).label("total_revenue")
).join(
    Repository, APICallLog.repo_id == Repository.id
).where(Repository.owner_id == current_user.id)
```

### 133.4 修复前后对比

| 指标 | 修复前 | 修复后 |
|-----|-------|-------|
| total_revenue | 100.0 (错误) | 0.41 (实际费用) |
| 数据来源 | bills 表 | api_call_logs 表 |

### 133.5 经验总结

1. **理解业务数据模型**：修改统计逻辑前先理解各表的业务含义
2. **字段类型问题**：字符串类型不能直接 SUM，需要 CAST
3. **收益来源**：总收益应从 API 调用记录计算
4. **关联计算**：owner 的收益需要通过 `Repository.owner_id` 关联计算

## 问题138：开发者用户判断逻辑不合理

### 138.1 问题描述

**发现时间**：2026-04-23

测试一般用户时，发现 `developer@example.com` 显示已是开发者，但实际上：
- 试用金额未领取
- 充值未成功
- 充值失败后仍被判断为开发者

**用户数据**：
- role: "user"（不匹配）
- user_type: "developer"（不匹配）
- API Keys: 1
- 仓库数: 0
- 成功充值记录: 5条

### 138.2 问题原因分析

**根本原因**：判断开发者身份的逻辑过于简单，只检查 `role` 或 `user_type` 其中一个字段。

**原代码逻辑**：
```python
# user.py 原有的 is_developer 判断
is_developer = user.role == "developer" or user.user_type == "developer"
```

**问题**：
1. 用户 `role="user"` 但 `user_type="developer"`，数据不一致
2. 只要有充值记录（包括失败的）就被视为开发者
3. 没有综合考虑多个指标（API Keys、仓库、充值记录等）

### 138.3 解决方案

**新增函数** `_is_real_developer()` 综合评估开发者身份：

```python
async def _is_real_developer(db: AsyncSession, user: User) -> dict:
    """
    综合评估用户是否为真正的开发者
    
    评估维度：
    1. 有仓库 → 真正的开发者
    2. role + user_type 都匹配 + 有 API Keys → 真正的开发者
    3. 有成功的充值记录 (bill_type='recharge' AND status='completed') → 真正的开发者
    4. role + user_type 都匹配 → 真正的开发者
    5. 只有其中一个匹配 → 不是真正的开发者
    """
    reasons = []
    is_developer = False
    
    # 1. 检查仓库
    repo_count = await db.scalar(
        select(func.count(Repository.id)).where(Repository.owner_id == user.id)
    )
    if repo_count and repo_count > 0:
        reasons.append(f"拥有{repo_count}个仓库")
        return {"is_developer": True, "reasons": reasons}
    
    # 2. 检查 role 和 user_type 是否一致
    role_match = user.role == "developer"
    user_type_match = user.user_type == "developer"
    
    # 3. 检查 API Keys
    api_key_count = await db.scalar(
        select(func.count(APIKey.id)).where(APIKey.user_id == user.id)
    )
    
    # 4. 检查成功的充值记录
    success_recharge = await db.scalar(
        select(func.count(Bill.id)).where(
            Bill.user_id == user.id,
            Bill.bill_type == "recharge",
            Bill.status == "completed"
        )
    )
    
    # 综合判断逻辑
    if role_match and user_type_match and api_key_count and api_key_count > 0:
        reasons.append("role和user_type匹配且拥有API Keys")
        is_developer = True
    elif success_recharge and success_recharge > 0:
        reasons.append(f"有{success_recharge}笔成功充值记录")
        is_developer = True
    elif role_match and user_type_match:
        reasons.append("role和user_type匹配")
        is_developer = True
    
    return {"is_developer": is_developer, "reasons": reasons}
```

### 138.4 开发者判断优先级

| 优先级 | 条件 | 结果 | 说明 |
|--------|------|------|------|
| 1 | 有仓库 | True | 真正的开发者 |
| 2 | role + user_type 都匹配 + 有 API Keys | True | 真正的开发者 |
| 3 | 有成功充值记录 | True | 真正的开发者 |

**问题**：部分用户 `role` 和 `user_type` 不一致。

**原因**：历史数据或迁移问题导致。

```python
# scripts/fix_user_role_consistency.py
async def fix_user_roles():
    """修复用户角色数据不一致问题"""
    db = AsyncSessionLocal()
    try:
        # 查找不一致的用户
        users = await db.execute(
            select(User).where(User.role != User.user_type)
        )
        for user in users.scalars():
            # 统一 user_type 为 role
            user.user_type = user.role
            logger.info(f"修复用户 {user.email}: user_type {user.user_type} -> {user.role}")
        await db.commit()
    finally:
        await db.close()
```

### 142.7 PermissionService 详细实现

```python
# src/services/permission_service.py
"""
统一权限检查服务
替代散落的硬编码权限检查
"""

from enum import Enum
from typing import List, Optional
from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy import select
from src.models import User, Repository


class UserRole(str, Enum):
    """用户角色枚举"""
    USER = "user"           # 普通用户
    DEVELOPER = "developer"  # API开发者
    ADMIN = "admin"         # 管理员
    SUPER_ADMIN = "super_admin"  # 超级管理员


# 默认权限配置
DEFAULT_PERMISSIONS = {
    "user": [
        "billing:read",
        "repo:read",
    ],
    "developer": [
        "billing:read",
        "billing:recharge",
        "repo:read",
        "repo:write",
        "api:read",
        "api:write",
        "api:delete",
        "log:read",
    ],
    "admin": ["*"],  # 所有权限
    "super_admin": ["*"],  # 所有权限
}


class PermissionService:
    """统一权限检查服务"""

    @staticmethod
    def get_user_role(user: User) -> str:
        """
        获取统一角色标识
        
        优先使用 user_type，fallback 到 role
        
        Args:
            user: 用户对象
            
        Returns:
            str: 角色标识
        """
        valid_roles = [r.value for r in UserRole]
        
        # 优先使用 user_type
        if user.user_type in valid_roles:
            return user.user_type
        
        # Fallback 到 role
        if user.role in valid_roles:
            return user.role
        
        # 默认返回 user
        return UserRole.USER.value

    @staticmethod
    def is_admin(user: User) -> bool:
        """检查是否为管理员"""
        role = PermissionService.get_user_role(user)
        return role in [UserRole.ADMIN.value, UserRole.SUPER_ADMIN.value]

    @staticmethod
    def is_super_admin(user: User) -> bool:
        """检查是否为超级管理员"""
        role = PermissionService.get_user_role(user)
        return role == UserRole.SUPER_ADMIN.value

    @staticmethod
    def is_developer(user: User) -> bool:
        """检查是否为开发者"""
        role = PermissionService.get_user_role(user)
        return role in [
            UserRole.DEVELOPER.value,
            UserRole.ADMIN.value,
            UserRole.SUPER_ADMIN.value
        ]

    @staticmethod
    def can_create_repo(user: User) -> bool:
        """检查是否可以创建仓库"""
        return PermissionService.is_developer(user)

    @staticmethod
    def has_permission(user: User, permission: str) -> bool:
        """
        检查是否拥有指定权限
        
        Args:
            user: 用户对象
            permission: 权限标识
            
        Returns:
            bool: 是否有权限
        """
        role = PermissionService.get_user_role(user)
        role_perms = DEFAULT_PERMISSIONS.get(role, [])
        
        # admin 和 super_admin 拥有所有权限
        if "*" in role_perms:
            return True
        
        return permission in role_perms
```

## 问题144：AsyncEngine 异步上下文管理器错误 (2026-04-23)

### 144.1 问题描述

**发现时间**：2026-04-23

运行 V4.0 迁移测试脚本时，部分测试失败，错误信息如下：

```
[Migration] ERROR: Migration failed: 'AsyncEngine' object does not support the asynchronous context manager protocol
```

**失败的测试步骤**：
- Step 1: 数据库迁移 ❌
- Step 2: 注册 Developer 用户 ✅
- Step 3: Owner 用户登录 ❌
- Step 4: 权限检查 ❌
- Step 5: 仓库创建权限 ❌

### 144.2 问题原因分析

**错误信息解读**：`AsyncEngine` 对象不支持异步上下文管理器协议。

**错误代码**：
```python
# 错误的写法
async with create_async_engine(self.database_url, echo=False) as engine:
    async_session = sessionmaker(bind=engine, class_=AsyncSession, expire_on_commit=False)
    async with async_session() as session:
        # ...
```

**原因**：
- SQLAlchemy 的 `create_async_engine()` 返回的是 `AsyncEngine` 对象
- `AsyncEngine` 不支持 `async with` 语法（不像同步引擎支持 `with`）
- 应该使用 `engine.begin()` 或 `engine.connect()` 获取连接

### 144.3 解决方案

**正确写法**：
```python
# 方法1：使用 engine.begin()（推荐）
engine = create_async_engine(self.database_url, echo=False)
async_session = sessionmaker(bind=engine, class_=AsyncSession, expire_on_commit=False)

async with engine.begin() as conn:
    async with async_session(bind=conn) as session:
        # 执行数据库操作
        await session.execute(query)
        await session.commit()

# 不要忘记关闭引擎
await engine.dispose()

# 方法2：使用 engine.connect()
engine = create_async_engine(self.database_url, echo=False)
async_session = sessionmaker(bind=engine, class_=AsyncSession, expire_on_commit=False)

async with engine.connect() as conn:
    async with async_session(bind=conn) as session:
        # ...
        
await engine.dispose()
```

### 144.5 修复示例

**修复前**（test_migration_v4.py）：
```python

```

**修复后**：
```python

    try:
        engine = create_async_engine(self.database_url, echo=False)
        async_session = sessionmaker(bind=engine, class_=AsyncSession, expire_on_commit=False)
        
        async with engine.begin() as conn:
            async with async_session(bind=conn) as session:
                # ...
```

### 144.6 经验总结

1. **SQLAlchemy 异步引擎使用规范**
   - `create_async_engine()` 创建的是 `AsyncEngine`
   - `AsyncEngine` 不支持 `async with`，但支持 `engine.begin()` 和 `engine.connect()`
   - `engine.begin()` 会自动管理事务（commit/rollback）
   - `engine.connect()` 需要手动管理事务

2. **资源管理**
   - 异步引擎使用完后需要调用 `await engine.dispose()` 释放资源
   - 使用 `async with` 确保连接正确关闭

3. **sessionmaker 绑定**
   - 需要将 `async_session` 绑定到 `conn`，而不是直接创建 session
   - 正确写法：`async_session(bind=conn)`

## 问题145：用户升级为开发者后界面不刷新 (2026-04-23)

### 145.1 问题描述

**发现时间**：2026-04-23

用户在普通用户页面点击"领取20元"或"升级为开发者"后：
- 提示"升级成功"
- 但页面没有自动刷新到开发者界面
- 手动刷新后仍然显示普通用户界面

### 145.2 问题原因分析

**问题出在**：用户状态更新与页面刷新的时序问题

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  问题流程                                                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 用户点击"升级为开发者"                                                  │
│         ↓                                                                  │
│  2. API 调用成功，服务器返回 user_type='developer'                           │
│         ↓                                                                  │
│  3. 前端调用 setAuth(user_data, ...) 更新 zustand store                    │
│         ↓                                                                  │
│  4. zustand persist 中间件 异步 写入 localStorage                          │
│         ↓                                                                  │
│  5. 【问题】window.location.reload() 立即执行                               │
│         ↓                                                                  │
│  6. localStorage 还没写入完成，页面就刷新了                                 │
│         ↓                                                                  │
│  7. 页面重新加载，从 localStorage 读取旧数据 (user_type='user')            │
│         ↓                                                                  │
│  8. 结果：用户看到的是旧界面                                                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**zustand persist 工作原理**：
- `persist` 中间件使用 `setTimeout` 或 `requestIdleCallback` 延迟写入 localStorage
- 这个延迟通常在 50-100ms 左右
- 如果在写入完成前刷新页面，就会丢失更新

### 145.3 解决方案

**新增函数** `refreshAuthAndReload`：

```typescript
/**
 * 刷新认证状态并等待持久化完成
 * 确保 localStorage 更新后再刷新页面
 */
const refreshAuthAndReload = useCallback(async () => {
  try {
    // 1. 获取最新的用户信息
    const user = await authApi.me()
    
    // 2. 更新 auth store（persist 中间件会自动写入 localStorage）
    setAuth(user, accessToken || '', refreshToken || '')
    
    // 3. 等待一小段时间确保 localStorage 写入完成
    await new Promise(resolve => setTimeout(resolve, 200))
    
    // 4. 验证 localStorage 是否已更新（可选，用于调试）
    const storedData = localStorage.getItem('auth-storage')
    
    // 5. 刷新页面
    window.location.reload()
  } catch (error) {
    // 即使出错也尝试刷新页面
    window.location.reload()
  }
}, [accessToken, refreshToken, setAuth])
```

   - 需要等待后再刷新页面

   - 使用 `JSON.parse()` 解析存储的数据
   - 检查 `state.user.user_type` 是否为新值

### 147.1 问题描述

```python
@router.post("/upgrade-with-payment", response_model=BaseResponse[UpgradeResponse])
async def upgrade_with_payment(
    request: Request,
    current_user: User = Depends(get_current_user),
    db: AsyncSession = Depends(get_db),
):
    """
    付费1元升级为开发者 (V4.1)
    
    用户支付1元后自动升级为开发者
    """
    from src.services.account_service import AccountService
    
    # 1. 检查是否已经是开发者
    if current_user.role == "developer" or current_user.user_type == "developer":
        raise APIError("您已经是开发者了，无需重复升级")
    
    # 2. 获取用户账户余额
    account_service = AccountService(db)
    account = await account_service.get_or_create_account(str(current_user.id))
    balance = float(account.balance) if account and account.balance else 0
    
    # 3. 检查余额是否足够
    if balance < UPGRADE_FEE:
        raise APIError(
            f"余额不足，升级需要 ¥{UPGRADE_FEE:.2f}，"
            f"当前余额 ¥{balance:.2f}。请先充值后再升级。",
            code="INSUFFICIENT_BALANCE"
        )
    
    # 4. 扣除升级费用
    await account_service.deduct_balance(
        user_id=str(current_user.id),
        amount=UPGRADE_FEE,
        source_type="upgrade",
        description=f"付费升级：{current_user.role} → developer",
    )
    
    # 5. 升级用户角色
    current_user.role = "developer"
    current_user.user_type = "developer"
    current_user.permissions = _get_developer_permissions()
    
    # 6. 记录操作日志
    log = UserOperationLog(...)
    db.add(log)
    
    await db.commit()
    
    return BaseResponse(
        success=True,
        message=f"升级成功！已扣除 ¥{UPGRADE_FEE:.2f}，您现在是开发者了",
        data={...}
    )
```

### 147.12 经验总结

1. **付费门槛要适中**：1元足够低门槛但又能过滤误操作
2. **二次确认很重要**：让用户知道升级的代价
3. **余额预检查**：避免扣费失败的用户体验差
4. **完善的日志记录**：便于问题排查和审计分析
5. **扣费要原子性**：余额扣除和角色升级要在同一事务中
